> 主题：Prompt Injection、最小权限、数据泄露与人工确认。

# 背景

Agent 会处理用户文本、评论和文件等不可信内容。恶意文本可能要求模型忽略规则、泄露数据或调用危险 Tool，这就是 Prompt Injection 风险。

模型无法成为安全边界。即使系统 Prompt 明确禁止，攻击仍可能改变模型行为，因此权限必须由确定代码强制执行。

# 核心思想

所有模型输入、输出和工具参数都不可信。最小权限、纵深防御和默认拒绝适用于 Agent：模型只能看到完成任务所需数据，只能调用允许工具，写操作经过策略与确认。

数据权限在检索前执行，不能把全库数据交给模型后要求它保密。审计记录决策证据，但也必须避免保存新的敏感副本。

# 工作原理

攻击内容可能来自用户 Prompt、任务评论、附件或 Tool 返回。模型把它们都视为上下文，无法可靠区分“数据中的指令”。运行时因此需要工具白名单、参数校验、输出过滤和权限执行器。

敏感写操作采用两阶段：模型生成结构化计划，系统展示影响范围，用户确认后以短期确认令牌执行。令牌绑定用户、动作、资源和过期时间，防止被替换。

# 企业实践

- 不向模型提供密钥、Token、数据库连接和不必要个人信息。
- 检索在权限过滤后进行，并限制数量和字段。
- Tool 不提供任意代码、SQL、文件路径或网络访问。
- 高风险写入、批量操作和外部通知要求人工确认。
- 对 Prompt Injection、越权、数据外泄和拒绝服务建立攻击测试集。
- 记录模型与 Tool 决策，但对 Prompt 和结果脱敏、限期保留。

# TaskFlow Pro应用

评论和附件内容都标记为不可信数据。项目分析 Agent 只能读取发起者可见项目；任务拆解默认只生成草稿，确认后 TaskService 再创建任务。

Agent 无权调用删除项目 Tool。即使未来开放，也必须单独授权、展示影响并使用绑定资源的确认令牌和审计。

# 常见误区

- 依靠系统 Prompt 保证绝不越权。
- 先检索全库，再让模型只回答授权内容。
- Tool 参数通过验证就跳过业务授权。
- 记录完整 Prompt、Token 和用户文件到日志。
- 将模型拒绝文案当作安全控制。

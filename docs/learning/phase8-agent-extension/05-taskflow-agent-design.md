> 主题：TaskFlow AI Assistant 场景、工具、数据流与验收。

# 背景

项目积累任务、成员、截止日期和状态历史后，用户需要风险总结、任务拆解和周报。直接让模型读取数据库会绕过权限与业务规则，也难以审计来源和写入影响。

TaskFlow Agent 应建立在 V1/V2 稳定 Service 之上，作为受控能力扩展，而非另建一套 AI 业务后门。

# 核心思想

三个首批场景采用不同自治程度：风险分析组合只读工具；任务拆解先生成草稿再确认写入；周报以确定数据聚合加模型表达为主。

所有结论应能追溯到结构化数据快照。Agent 负责解释和组合，Service 负责事实、权限和写入。

# 工作原理

风险分析由 Project Tool 获取范围，Task Tool 获取逾期、阻塞和无人负责任务，Statistics Tool 计算完成率与负载，模型输出结构化风险等级、证据和建议。

任务拆解将自然语言需求转为候选任务 Schema，系统校验负责人、日期和依赖，用户确认后批量调用 TaskService。周报按固定时间窗查询变化，模型只负责组织 Markdown。

每次 Agent Run 记录发起者、项目、数据时间点、Tool 调用、Prompt/模型版本、结果和确认动作。长任务由 Celery 执行并支持状态查询。

# 企业实践

- 先为每个场景定义非 AI 基线和验收指标。
- 报告明确数据截止时间，缺失数据不得臆测。
- 任务草稿与已创建任务状态分离。
- 批量创建使用幂等请求 ID 和事务/部分失败策略。
- 评测覆盖事实准确、权限隔离、格式稳定和操作安全。
- 模型失败时保留原始结构化统计供用户查看。

# TaskFlow Pro应用

建议目录包括 `app/agents/` 场景编排、`app/tools/` 业务适配、`app/services/ai_*` 运行管理和 `app/tasks/` 异步 Job。具体结构在 Phase 8 设计时根据现有项目调整。

核心路径始终是：

`Authenticated User → Agent Run → Authorized Tool → Service → Repository → Database`

项目风险报告只读；任务拆解写入前确认；周报生成结果作为派生文档保存。所有功能可在关闭 AI 时不影响基础项目管理。

# 常见误区

- V1 业务尚不稳定就先让 Agent 直接写库。
- 报告没有数据时间范围和证据引用。
- 草稿生成后自动批量创建任务。
- 把自然语言周报当作唯一统计结果。
- AI 服务故障导致核心 API 无法启动。

# 学习检查问题

1. 三个 Agent 场景为何采用不同自治程度？
2. 风险报告怎样提供可追溯证据？
3. 任务拆解从草稿到写入需要哪些校验与确认？
4. 周报哪些步骤应确定化，哪些适合模型生成？
5. 如何证明 Agent 关闭后 TaskFlow 核心业务仍可用？

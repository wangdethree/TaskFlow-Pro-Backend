> 主题：LLM、Workflow、Agent、规划循环与适用边界。

# 背景

普通 LLM 只能基于输入生成文本，无法可靠读取实时项目数据或执行受控操作。固定 Workflow 可以编排已知步骤，但面对开放目标时缺乏动态选择。

Agent 在模型周围加入工具、状态和控制循环，使系统能够观察环境、选择动作并根据结果继续处理。它同时引入不确定性、成本和安全风险。

# 核心思想

Agent 的本质不是“会聊天”，而是模型驱动的决策循环：接收目标和上下文，选择有限工具，观察结构化结果，再决定下一步或结束。

能用确定代码解决的问题优先使用确定代码。只有任务路径难以预先枚举、需要自然语言推理或跨工具组合时，Agent 才可能提供价值。

# 工作原理

控制器向模型提供系统规则、用户目标、工具 Schema 和有限上下文。模型输出工具调用或最终结果；运行时验证参数、执行 Tool、把结果作为 Observation 返回，直到结束或达到步数/成本限制。

模型没有数据库真相，记忆也不是天然可靠状态。所有实时事实来自 Tool，持久状态进入受控存储。循环必须处理超时、重复调用、无效参数和模型无法完成等情况。

# 企业实践

- 先用 Workflow 建立可预测基线，再评估哪些节点需要 Agent 决策。
- 限制工具集合、最大步数、时间和成本。
- 输入输出结构化，关键结论附数据来源和时间范围。
- 写操作要求幂等和必要人工确认。
- 建立离线评测集，而非只靠几个演示对话。
- 明确失败降级，AI 不可用时核心后端继续工作。

# TaskFlow Pro应用

项目风险分析需要组合 Project、Task、Statistics Tool，适合受控 Agent；固定周报数据提取与模板生成可以先用 Workflow，未必需要完全自治。

Agent 是业务系统的可选上层，不能成为创建任务和权限判断的唯一入口。AI 请求通过 Celery 处理长耗时，结果保存为可重试派生数据。

# 常见误区

- 把一次 LLM 调用都称为 Agent。
- 为确定 CRUD 流程引入开放式 Agent。
- 把模型记忆当作权威数据库。
- 没有步数和成本限制。
- 演示一次成功就认为生产可靠。

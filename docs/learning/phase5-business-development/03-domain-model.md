> 主题：领域模型、实体、值对象、不变量与业务语言。

# 背景

以数据库表为中心开发容易把系统简化为 CRUD，真实规则散落在 if/else 中。产品说“项目负责人”“任务流转”，代码却只有字段更新，沟通与实现逐渐分离。

领域建模让代码表达业务概念、关系和不变量，使变化集中在对应模型与用例。

# 核心思想

实体由身份和生命周期区分，值对象由属性值定义。聚合用于界定一致性边界和修改入口，不意味着把全部关联数据一次加载。

领域语言应与需求一致；不变量在任何合法状态中都成立，例如任务负责人必须是项目成员。领域模型可以是行为丰富的对象，也可以由 Service 与规则对象承担，关键是规则有唯一权威位置。

# 工作原理

用例加载执行规则所需的实体和关系，调用领域行为验证状态并产生变化。持久化层保存结果，领域事件描述“已经发生的业务事实”，供通知或统计异步处理。

跨多个聚合的强一致操作成本高，需识别哪些规则必须同事务保证，哪些可以最终一致。模型边界来自业务一致性需求，而不是目录大小。

# 企业实践

- 与产品统一术语，避免同一概念多个名称。
- 区分业务不变量与展示规则。
- 不让 ORM 技术细节主导全部领域设计。
- 聚合保持适当大小，列表统计用查询模型而非加载整棵对象图。
- 领域事件使用过去式事实并具备唯一标识。
- 简单模块不强行引入复杂 DDD 战术模式。

# TaskFlow Pro应用

Project、ProjectMember、Task 是核心实体；任务状态、项目角色可建模为受控值。项目成员关系承载加入时间、角色和状态，不是无语义中间表。

“TaskCompleted”可作为业务事实触发通知和统计。状态转换规则集中在 Task 领域规则或 TaskService，不能由客户端任意写状态字段。

# 常见误区

- 把每张表都称为聚合根。
- 领域模型只是无行为的 ORM 字段容器。
- 所有一致性都要求一个巨大事务。
- 为简单枚举建立复杂框架。
- 领域事件在事务提交前直接调用外部系统。

# 学习检查问题

1. 实体和值对象如何区分？
2. 项目成员为什么是有业务含义的实体关系？
3. 聚合边界与事务边界有什么关系？
4. 哪些 TaskFlow 规则属于真正不变量？
5. 领域事件为什么描述事实而不是命令？

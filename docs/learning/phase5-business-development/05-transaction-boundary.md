> 主题：业务事务边界、外部副作用与最终一致性。

# 背景

数据库事务能保护单库操作，但业务用例还可能调用 Redis、MinIO、Celery 和邮件系统。把它们全部假设成一个原子操作不现实；把每一步单独提交又会留下半完成状态。

事务边界设计决定哪些不变量必须立即一致，哪些副作用允许稍后完成和补偿。

# 核心思想

本地事务覆盖数据库内需要共同成功的变化。外部系统通常通过“先提交业务事实，再可靠触发副作用”实现最终一致性。

事务边界应尽量短，但不能短到破坏业务原子性。可重试、幂等、Outbox 和补偿是跨边界可靠性的关键工具。

# 工作原理

Service 开启事务，完成数据校验与写入；提交后才对外可见。若需要可靠消息，可在同一事务写入 Outbox 记录，由独立发布器投递，成功后标记状态。

消费者可能至少一次收到消息，因此使用事件 ID、业务唯一键或状态检查实现幂等。外部对象存储失败时，可保留待处理状态或执行补偿清理。

# 企业实践

- 明确每个用例的原子数据集合。
- 不在持锁事务内等待邮件、LLM 或长文件上传。
- 事务提交与消息发布之间设计可靠桥梁。
- 重试有次数、退避和死信/人工处理策略。
- 补偿不是数据库 rollback，要定义业务可接受结果。
- 记录跨系统关联 ID 便于追踪。

# TaskFlow Pro应用

创建任务和任务初始审计记录可同事务提交；通知事件通过 Outbox/Celery 异步发送。若通知暂时失败，任务仍存在，系统重试通知而不是回滚任务。

文件上传可先生成上传状态，再写 MinIO 和元数据，失败后清理孤儿对象。AI 报告属于可重试派生结果，不应占用任务写事务。

# 常见误区

- 在数据库事务里同步调用 LLM。
- 先发送 Celery 消息再提交业务数据，Worker 读取不到记录。
- 认为 Celery 投递恰好一次。
- 失败后无限快速重试。
- 把所有外部失败都要求回滚已完成核心业务。

# 学习检查问题

1. 本地事务与分布式副作用的边界在哪里？
2. Outbox 如何缩小“提交成功但消息丢失”的窗口？
3. 至少一次投递为什么要求消费者幂等？
4. 文件上传失败可以有哪些补偿策略？
5. 哪些 TaskFlow 操作必须强一致，哪些可最终一致？

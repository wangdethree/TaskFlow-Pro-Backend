> 主题：有限状态机、状态转换、守卫条件与并发控制。

# 背景

若任务状态只是可写字符串，客户端可以从任意状态跳转，重复完成也可能重复发送通知。规则分散后，API、后台任务和 Agent 对合法状态的理解不同。

状态机把允许状态、转换和条件显式化，使生命周期可验证和测试。

# 核心思想

有限状态机由状态集合、事件、转换和守卫条件组成。事件表达意图，转换决定从当前状态能否到达目标状态，副作用由转换结果触发。

状态机适合存在有限、明确生命周期的对象；对简单独立布尔字段强行使用会增加复杂度。

# 工作原理

处理事件时读取当前状态，查找对应转换，验证权限和守卫条件，更新状态并记录版本或历史。非法转换返回业务冲突，不应静默覆盖。

并发请求可能都基于旧状态判断成功。乐观锁通过版本条件更新，影响行数为零则表示冲突；悲观锁则锁定行但降低并发并可能死锁。

# 企业实践

- 转换表或规则集中维护，API 只发送事件或目标意图。
- 权限与状态守卫都要测试。
- 状态历史记录操作者、时间、前后状态和原因。
- 副作用保证幂等，重试不重复通知。
- 状态新增评估旧数据、API 兼容和统计查询。
- 并发编辑频繁时使用版本号检测丢失更新。

# TaskFlow Pro应用

V1 的 Task 状态为 TODO、DOING、DONE。系统需要明确是否允许 DONE 回到 DOING、谁能转换、负责人为空能否开始、完成后如何通知。

Agent 提议完成任务时仍调用 TaskService 的转换方法，不能直接更新字段。状态版本和历史为后续风险分析提供可信数据。

# 常见误区

- 允许客户端直接提交任意状态字符串。
- 只校验目标状态存在，不校验来源状态。
- 状态修改成功后同步发送不可重试通知。
- 忽略并发覆盖，最后写入者静默获胜。
- 用状态机包装没有生命周期的简单属性。

2. 为什么“目标状态合法”不足以证明转换合法？
3. 两个用户同时完成任务会产生什么竞态？
4. 状态历史对审计和 Agent 分析有什么价值？
5. 哪些场景不值得引入状态机？

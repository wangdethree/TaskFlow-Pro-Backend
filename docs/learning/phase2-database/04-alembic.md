> 主题：Alembic 数据库迁移、版本图与安全演进。

# 背景

数据库结构会随功能变化。手工在每台环境执行 SQL 无法证明执行顺序、遗漏情况和回滚路径，也会让代码版本与 Schema 不匹配。

迁移工具把结构变化纳入版本控制，使开发、测试和生产能够沿同一版本图演进。

# 核心思想

迁移是数据库 Schema 的可执行历史。每个 revision 描述从一个已知版本到下一个版本的变化，`upgrade` 前进，`downgrade` 表达可行的回退。

自动生成只负责比较元数据与数据库，不理解数据迁移、线上锁表和业务含义；迁移文件必须人工审查。

# 工作原理

Alembic 用版本表记录数据库当前 revision。迁移脚本通过 `revision` 和 `down_revision` 构成版本图，运行时计算从当前节点到目标节点的路径并执行操作。

Autogenerate 比较 SQLAlchemy MetaData 与数据库反射结果，生成候选差异。重命名经常被识别为删除加新增，可能造成数据丢失，需要人工改写。

DDL 的事务能力和锁行为取决于数据库。即使迁移工具报告成功，也要确认数据回填、索引创建和应用兼容窗口。

# 企业实践

- 迁移文件与 Model 改动在同一功能变更中提交。
- 在空库和带历史数据的数据库分别验证升级。
- 采用扩展—迁移—收缩策略处理无停机字段变化。
- 大表加索引、改列和回填要评估锁、时长和失败恢复。
- 不在应用每个 Worker 启动时竞争执行迁移。
- 已发布迁移不随意重写，通过新 revision 修正。

# TaskFlow Pro应用

用户、项目、成员和任务表通过初始 revision 创建。后续添加任务版本号、通知索引或文件元数据，都必须有独立迁移，并在 Docker 部署流程中由单一迁移步骤执行。

种子角色与权限属于数据初始化问题，需要明确由迁移、初始化命令还是部署任务负责，避免每次应用启动重复写入。

# 常见误区

- 完全信任 autogenerate，不阅读迁移脚本。
- 修改 Model 后只在本地直接改表。
- 把重命名列当作删除再新建。
- 认为有 downgrade 就一定能无损回滚数据。
- 多个应用实例启动时同时执行迁移。

# 学习检查问题

1. revision 图如何决定迁移顺序？
2. autogenerate 为什么无法理解列重命名？
3. 数据迁移与 Schema 迁移有何不同风险？
4. 扩展—迁移—收缩怎样支持兼容发布？
5. TaskFlow 的角色种子数据应如何避免重复初始化？

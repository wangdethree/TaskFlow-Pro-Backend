> 主题：MySQL 关系建模、约束、索引与数据生命周期。

# 背景

业务数据不仅要被保存，还要在并发、失败和长期演进中保持正确。只按接口字段建表，容易出现重复成员、无效负责人、孤儿评论和列表查询退化。

关系数据库通过表、键、约束和事务建立可验证的数据模型。企业数据库设计的重点是把业务不变量落到可靠边界，而不是先建表再让代码补救全部错误。

# 核心思想

关系建模把实体、关系和约束显式化。主键标识事实，外键保护引用，唯一约束保护不可重复关系，索引服务于真实查询路径。

规范化减少重复和更新异常；适度反规范化可以优化已证实的读取热点，但必须承担同步成本。数据库约束与 Service 校验互补：前者抵御并发和绕过，后者提供业务语义与友好错误。

# 工作原理

InnoDB 以聚簇索引组织主键数据，二级索引叶节点保存主键值，因此过宽、随机的主键会影响存储和写入。联合索引的列顺序决定可支持的过滤与排序，最左前缀和选择性需要结合查询分析。

外键在写入时验证引用，唯一索引在并发情况下最终保证不重复。事务隔离与锁影响多个会话看到的数据和相互等待行为。

执行计划展示优化器选择的访问路径。索引不是越多越好：它占空间，并增加插入、更新和删除成本。

# 企业实践

- 先写业务不变量和主要查询，再设计表与索引。
- 关联关系有属性时使用关联实体，如项目成员包含角色、加入时间和状态。
- 明确时间、时区、字符集、排序规则和金额精度。
- 外键删除策略要与归档、审计和合规要求一致。
- 默认列表分页，观察执行计划和慢查询后再优化。
- Schema 变更通过 Alembic，生产大表变更评估锁与回滚。

# TaskFlow Pro应用

用户邮箱需要唯一约束；`project_members` 对项目与用户组合唯一，并保存项目角色；任务负责人引用用户，但 Service 还要保证其属于目标项目。

任务列表常按项目、状态、负责人和创建时间过滤，需要根据实际查询设计联合索引。项目删除更适合先归档，避免级联删除破坏评论、通知和审计历史。

# 常见误区

- 只在代码中查重，不建立唯一约束。
- 每个字段都建索引，忽略写放大。
- 用字符串保存所有状态却没有受控枚举或校验。
- 看到查询慢就先反规范化，不分析执行计划。
- 默认级联删除所有关联数据。

# 学习检查问题

1. 为什么项目成员应是关联实体而不只是普通多对多表？
2. Service 已查重后为什么仍需唯一约束？
3. 联合索引列顺序应依据什么决定？
4. 任务负责人外键为何不能保证其属于同一项目？
5. 项目删除与归档分别影响哪些数据生命周期要求？

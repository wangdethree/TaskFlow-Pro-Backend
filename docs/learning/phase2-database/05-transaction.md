> 主题：ACID、隔离级别、锁、并发异常与事务设计。

# 背景

创建项目可能同时写项目、创建者成员关系和审计信息。中途失败却已提交部分数据，会产生无法解释的状态。并发请求还会造成重复成员、丢失更新和库存式竞争。

事务把一组数据库操作定义为一致性单元，并通过隔离机制控制并发可见性。

# 核心思想

ACID 中，原子性保证整体成功或回滚；一致性依赖约束和业务规则；隔离性控制并发事务相互可见；持久性保证提交后的数据可靠保存。

事务边界应对应业务不变量，而不是单个 Repository 方法。事务越长，锁与连接占用越大；越短则可能无法保护完整规则，需要在正确性和并发间取舍。

# 工作原理

数据库通过 undo/redo、MVCC 和锁实现事务能力。隔离级别决定脏读、不可重复读和幻读等现象能否出现，具体行为还受 MySQL 实现和查询锁定方式影响。

普通读与锁定读不同。更新同一行会竞争排他锁；多个事务按不同顺序获取资源可能死锁，数据库会回滚其中一个事务，应用需要有限重试。

唯一约束是并发查重的最终防线。先查询再插入仍有竞态，必须捕获约束错误并转换为稳定业务结果。

# 企业实践

- Service 定义用例事务，Repository 不擅自提交。
- 外部 HTTP、邮件和 AI 调用不放在长数据库事务中。
- 对并发冲突使用唯一约束、乐观锁或显式锁，而非只靠内存判断。
- 统一资源访问顺序降低死锁概率，并对可重试错误有限重试。
- 事务失败后回滚，日志记录业务标识而非敏感数据。
- 明确“数据库提交成功但消息发送失败”的后续一致性方案。

# TaskFlow Pro应用

创建项目与创建者成员关系必须同一事务。分配任务时需验证负责人仍是项目成员，并考虑验证后成员被移除的并发窗口。

任务状态更新可引入版本号做乐观锁，防止两个编辑者互相覆盖。通知发送应在业务提交后异步处理，不能为了调用外部服务长期占用数据库事务。

# 常见误区

- 每条 SQL 都立即 commit，失去业务原子性。
- 事务包围远程调用，导致连接与锁长期占用。
- 先查重再插入，却没有唯一约束。
- 所有并发问题都用最高隔离级别解决。
- 捕获数据库异常后不 rollback 继续使用 Session。

# 学习检查问题

1. 数据库一致性为何不只由数据库自动保证？
2. 先查询后插入为什么仍可能重复？
3. 事务边界为什么应由 Service 看见完整用例？
4. 乐观锁适合 TaskFlow 哪类更新？
5. 业务提交成功但通知消息发送失败应如何思考？

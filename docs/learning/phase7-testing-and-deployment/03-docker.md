> 主题：Docker 镜像、容器、分层构建与运行边界。

# 背景

虚拟环境只隔离 Python 包，无法固定操作系统库、用户、文件系统和启动命令。开发机能运行的依赖可能在服务器缺少系统库。

Docker 用镜像描述运行文件系统和元数据，用容器提供可重复进程环境，减少“机器差异”带来的部署问题。

# 核心思想

镜像是只读分层模板，容器是镜像的运行实例。构建应确定、最小且可审查；运行时配置和密钥不写入镜像。

容器不是虚拟机，也不是安全沙箱的全部。进程、权限、网络和资源限制仍需显式设计。

# 工作原理

Dockerfile 每个构建步骤产生可缓存层。改动靠前层会使后续缓存失效，因此通常先复制依赖声明并安装，再复制应用代码。

容器使用联合文件系统和隔离机制运行进程；可写层随容器删除，持久数据应使用 Volume 或外部服务。PID 1 需要正确接收信号，应用才能优雅关闭。

# 企业实践

- 固定基础镜像版本并定期更新安全补丁。
- 使用非 root 用户，减少镜像内容和构建上下文。
- 不复制 `.env`、`.git`、`.venv` 和缓存进入镜像。
- 使用多阶段构建或 Wheel 降低运行镜像工具链。
- 健康检查区分进程存活与服务就绪。
- 容器日志输出到标准输出，由平台收集。

# TaskFlow Pro应用

后端镜像固定 Python 3.11，安装锁定依赖后复制 `backend`。Uvicorn/FastAPI 是 Web 容器主进程，Celery 使用同一应用镜像但不同启动命令。

MySQL 数据不写入后端容器层。Alembic 迁移由单独部署步骤执行，不能让每个 Web 容器抢跑。

# 常见误区

- 把开发 `.venv` 复制进 Linux 镜像。
- 使用 `latest` 基础镜像且不记录实际版本。
- 密钥写进 Dockerfile 或镜像层。
- 容器内以 root 运行所有服务。
- 数据库数据保存在容器可写层。

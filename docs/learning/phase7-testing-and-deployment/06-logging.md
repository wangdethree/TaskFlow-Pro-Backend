> 主题：结构化日志、关联 ID、级别、脱敏与可观测性。

# 背景

线上故障无法靠断点复现。没有上下文的 `print` 很难把一次请求跨 API、数据库和 Celery 的行为串联，也可能意外泄露密码和 Token。

日志是系统产生的事件证据，需要可搜索、可关联、可控制成本，并与指标和追踪互补。

# 核心思想

结构化日志把时间、级别、事件、Request ID、用户/项目安全标识等作为字段，而不是拼接不可解析文本。日志记录事实和决策，不记录所有数据。

可观测性通常由日志、指标和追踪组成：日志解释具体事件，指标展示趋势，追踪连接分布式调用。仅增加日志数量不会自动提高可观测性。

# 工作原理

请求入口生成或接收关联 ID，放入上下文并随下游调用、Celery 消息传播。Logger 根据级别和 Handler 输出结构化记录，部署平台收集、索引和保留。

异常日志应包含堆栈和内部定位信息，但外部响应只返回安全错误与关联 ID。异步任务使用 task ID、event ID 连接生产与消费记录。

# 企业实践

- 定义事件名称和字段规范，避免自由文本失控。
- INFO 记录关键生命周期，WARNING 表示可恢复异常，ERROR 表示失败结果。
- 密码、Token、Cookie、密钥和敏感文件名进行禁止或脱敏。
- 避免每条数据库记录打 INFO，控制成本和噪声。
- 使用日志采样和保留策略，但安全审计单独满足要求。
- 日志时间统一、字段稳定，并与告警指标连接。

# TaskFlow Pro应用

API 日志包含 Request ID、路由、方法、状态和耗时；权限拒绝记录主体、权限代码和资源摘要；Celery 记录 task/event ID 和重试次数。

Agent 调用额外记录 Tool 名称、模型、耗时、Token 用量和结果状态，但不记录完整敏感 Prompt。审计日志与调试日志分开治理。

# 常见误区

- 捕获异常后只打印文案，不保留堆栈。
- 记录完整请求 Body 和 Authorization Header。
- 所有事件都用 INFO 或 ERROR。
- 用日志替代指标和审计表。
- Request ID 只存在 API，不传播到任务队列。

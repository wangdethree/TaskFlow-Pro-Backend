> 主题：pytest 收集、Fixture、参数化与异步测试。

# 背景

手工调用接口只能验证少量当前路径，无法在每次修改后稳定回归权限、事务和异常场景。pytest 提供可组合 Fixture、发现机制和丰富断言，让测试成为可执行规格。

测试工具的价值不在语法简短，而在建立可重复、隔离且能定位失败的反馈系统。

# 核心思想

测试遵循 Arrange、Act、Assert，并控制外部状态。Fixture 描述测试依赖及生命周期，参数化让同一规则覆盖多组输入。

测试应验证可观察行为，而非绑定内部实现细节。越接近数据库和 HTTP 的测试成本越高，但能发现纯 Mock 无法发现的问题。

# 工作原理

pytest 按规则收集测试模块与函数，解析函数参数对应 Fixture，构建依赖图并按 scope 创建、缓存和清理资源。Fixture 的 `yield` 后部分负责 teardown。

插件支持异步测试和 HTTP 客户端。异步测试必须在受控事件循环中运行，数据库 Fixture 要确保每个测试数据隔离和连接释放。

# 企业实践

- Fixture 小而可组合，避免一个万能 Fixture 隐式创建大量数据。
- 测试名表达场景和预期，失败后无需读实现才知道意图。
- 参数化覆盖角色、状态转换和边界值。
- 固定时间、随机数和外部服务，降低脆弱性。
- 测试数据库使用与生产相同数据库类型验证方言差异。
- CI 输出失败证据，不打印凭证和生产数据。

# TaskFlow Pro应用

认证 Fixture 提供不同角色用户和 Token；项目 Fixture 构造跨项目场景；数据库 Fixture 管理测试 Session。权限矩阵用参数化覆盖 Admin、Manager、Member 的允许与拒绝。

Celery、Redis 和 MinIO 分别有单元替身与集成测试，不把所有外部系统都 Mock 后声称链路已验证。

# 常见误区

- Fixture 之间隐式依赖过深。
- 测试执行顺序相关，共享残留数据。
- 只断言状态码，不检查数据库副作用。
- Mock 掉被测试的核心业务规则。
- 异步资源未关闭，产生偶发警告和失败。

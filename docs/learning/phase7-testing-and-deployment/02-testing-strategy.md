> 主题：测试金字塔、风险覆盖、契约与端到端策略。

# 背景

只有单元测试无法验证数据库约束和路由装配；只有端到端测试又慢、难定位且维护成本高。企业需要按风险组合不同层次测试。

测试策略决定测试什么、在哪一层测试以及失败时能提供什么信心，而不是追求单一覆盖率数字。

# 核心思想

测试金字塔强调大量快速单元测试、适量集成测试和少量关键端到端测试。比例不是固定公式，应由系统风险决定。

优先覆盖高影响、高变化和历史易错部分：权限边界、事务回滚、状态机、迁移与消息幂等。代码覆盖率只能发现未执行区域，不能证明断言有效。

# 工作原理

单元测试隔离 Service 规则；集成测试连接真实 MySQL/Redis 等组件；API 测试通过 ASGI 客户端验证路由、依赖和序列化；端到端测试在部署形态验证完整链路。

契约测试验证 API 或消息 Schema 的兼容性。迁移测试从旧版本数据库升级，检查结构和数据结果。

# 企业实践

- 从验收标准和故障模式设计测试，不从函数列表机械生成。
- 每个缺陷修复先增加能复现问题的测试。
- 正向、负向、边界和并发场景都有代表覆盖。
- 将快速检查放在提交前，完整集成测试放入 CI。
- 测试数据最小化且意图清楚。
- 对脆弱测试追根因，不简单增加重试掩盖竞态。

# TaskFlow Pro应用

Service 单测覆盖任务转换；MySQL 集成测试覆盖唯一约束和事务；API 测试覆盖认证链；跨项目越权必须作为高风险负向场景。

Docker Compose 冒烟测试验证迁移、健康检查、Celery 和外部服务连接。Agent 阶段增加 Tool 契约、权限透传和危险写操作确认测试。

# 常见误区

- 追求 100% 覆盖率却没有有效断言。
- 所有测试都走完整 HTTP 和数据库，反馈过慢。
- 单元测试完全复制实现步骤。
- 忽略迁移和配置测试。
- 将偶发失败统一标记为重试。

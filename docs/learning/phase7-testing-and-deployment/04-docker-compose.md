> 主题：Docker Compose 服务编排、网络、依赖与健康检查。

# 背景

TaskFlow 需要 Web、MySQL、Redis、Celery、MinIO 和 Nginx。逐个手工启动难以保证网络、配置和依赖顺序一致，也增加新成员上手成本。

Compose 用声明文件描述多容器开发和小规模部署拓扑，使环境可重复启动和清理。

# 核心思想

Compose 管理服务而不是把所有进程塞进一个容器。每个服务有独立镜像、配置、网络、健康检查和持久化需求。

“容器已启动”不等于“应用已就绪”。依赖顺序只能辅助启动，服务仍需重试并容忍下游暂时不可用。

# 工作原理

Compose 创建项目级网络，服务通过服务名 DNS 发现，例如后端连接 `mysql` 而非容器 IP。Volume 保存数据库和对象数据，环境变量注入运行配置。

`depends_on` 表达启动依赖，在结合健康条件时可等待基础服务健康，但不能处理运行期间故障。健康检查应检测最小关键能力并有合理间隔和超时。

# 企业实践

- 基础配置版本化，密钥通过外部环境或 Secret 注入。
- 开发与生产差异用明确覆盖文件或部署平台管理。
- 为 MySQL、Redis、MinIO 和应用设置健康检查。
- 服务限制资源并配置重启策略，但不让无限重启掩盖配置错误。
- 数据 Volume 有备份和清理规则。
- CI 使用干净 Compose 项目名避免并行冲突。

# TaskFlow Pro应用

Compose 将编排 `api`、`worker`、`mysql`、`redis`、`minio` 和 `nginx`。API 与 Worker 复用应用镜像，通过服务名访问基础设施。

迁移可作为一次性服务先运行；健康接口验证应用就绪。开发环境可只暴露必要端口，内部数据库不必对外网开放。

# 常见误区

- 在容器配置中连接 `localhost` 寻找其他服务。
- 认为 `depends_on` 保证数据库永远可用。
- 把所有密钥直接提交到 Compose 文件。
- 没有 Volume 却期望重建容器后数据存在。
- 生产直接使用开发端口和调试配置。

# 学习检查问题

1. Compose 网络中服务为何通过服务名访问？
2. 启动顺序、健康和运行期可用性有什么区别？
3. API 与 Worker 为什么适合复用镜像？
4. 哪些数据需要 Volume，哪些应保持无状态？
5. 如何防止 CI 并行 Compose 环境互相干扰？

> 主题：JWT 结构、签名、声明和 Token 生命周期。

# 背景

分布式客户端需要在后续请求中携带已认证身份。服务端 Session 易于撤销但需要集中状态；JWT 让接收方通过签名验证声明，减少每次向会话存储查询。

JWT 解决的是受保护声明传递，不自动解决退出、设备管理、权限实时变化和 Token 泄露。

# 核心思想

JWT 由 Header、Payload 和 Signature 组成。Payload 通常只编码而不加密，客户端可读取；签名保证内容未被篡改和来自持有密钥的一方。

Access Token 应短期、最小化，只携带稳定身份和必要声明。动态权限不宜完整塞入长寿命 Token，否则角色变更后旧权限持续有效。

# 工作原理

签发方对 Base64URL 编码的 Header 和 Payload 计算签名。验证方必须固定允许的算法，校验签名以及 `exp`、`nbf`、`iss`、`aud`、`sub`、`jti` 等适用声明。

对称算法共享签发和验证密钥；非对称算法用私钥签发、公钥验证，适合多服务验证但增加密钥轮换复杂度。验证签名不代表用户仍启用，因此通常仍需账户状态检查。

# 企业实践

- Token 仅通过 HTTPS 传输，密钥放在安全配置中并支持轮换。
- 明确 Access/Refresh Token 边界、有效期和重放风险。
- 固定算法白名单，不信任 Token Header 自行选择算法。
- 预留时钟偏差但保持有限窗口。
- 高风险会话用 `jti`、会话版本或 Refresh Token 轮换实现撤销。
- 不在 Payload 放密码、隐私或机密配置。

# TaskFlow Pro应用

Access Token 的 `sub` 标识用户，认证依赖验证 Token 后查询当前用户状态。项目角色从数据库或短期权限缓存读取，避免 Token 长期携带过时权限。

Redis 可保存撤销 `jti`、用户会话版本或 Refresh Token 状态。退出、禁用和密码修改时按风险策略使旧凭证失效。

# 常见误区

- 认为 JWT Payload 已加密。
- Token 有签名就不查询用户状态。
- 使用永不过期 Token 省去续期设计。
- 把全部 RBAC 权限塞入长期 Token。
- 把 Refresh Token 当作普通 Access Token 使用。

> 主题：认证与授权、身份生命周期和可信边界。

# 背景

后端必须先判断请求者是谁，才能执行项目成员和任务权限规则。只依赖客户端提交用户 ID 或隐藏按钮，会让攻击者伪造身份或直接调用接口。

认证建立可信身份，授权在此基础上决定行为。二者混合会导致登录逻辑散落在业务代码中，也容易把“已登录”误认为“有权操作”。

# 核心思想

认证是把外部凭证映射为系统主体的过程；授权判断该主体对目标资源是否拥有操作权限。会话、Token 只是传递认证状态的机制，不是身份本身。

身份生命周期包括注册、凭证验证、会话签发、续期、退出、密码修改、禁用和审计。任何状态变化都可能要求旧凭证失效。

# 工作原理

登录时服务端根据稳定标识查询用户，验证密码哈希和账户状态，成功后签发受保护的会话凭证。后续请求携带凭证，认证组件校验完整性、有效期和主体声明，再从数据库获得当前账户状态。

客户端输入的 `user_id` 不能作为可信身份；真正身份来自验证后的凭证。认证失败应统一响应，避免通过差异信息枚举账号。

# 企业实践

- 登录、找回密码和敏感操作设置速率限制与审计。
- 账号禁用、密码重置和角色变更要有会话失效策略。
- 日志不记录密码、完整 Token 或验证码。
- 使用 HTTPS 保护传输中的凭证。
- 认证组件只建立主体，项目级授权交给权限层。
- 对自动化账号、用户账号和 Agent 调用明确不同凭证范围。

# TaskFlow Pro应用

注册和登录建立 User 身份；`get_current_user` 将有效凭证转换为当前用户。项目、任务、文件等 Service 再基于成员关系和权限做授权。

V1 可使用短期 Access Token，V2 引入 Redis 会话版本或撤销状态，使禁用用户和修改密码后能够主动阻止旧 Token。

# 常见误区

- 已登录就允许访问全部项目。
- 接受请求 Body 中的用户 ID 作为操作者。
- 登录失败分别返回“用户不存在”和“密码错误”。
- 退出登录只删除客户端 Token，却声称服务端已撤销。
- 在日志中记录 Authorization Header 方便排错。

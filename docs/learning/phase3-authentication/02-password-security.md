> 主题：密码哈希、盐、工作因子和凭证防护。

# 背景

数据库可能因漏洞、备份泄露或权限错误被读取。明文或可逆加密密码一旦泄露，攻击者可直接登录，并利用用户跨站复用密码攻击其他系统。

密码存储必须让服务端能够验证“输入是否正确”，同时不能还原原始密码，并提高离线猜测成本。

# 核心思想

密码使用专门的慢哈希/KDF，而不是普通快速摘要。每个密码配独立随机盐，阻止相同密码产生相同结果和预计算彩虹表；工作因子控制单次猜测成本。

安全参数需要随硬件演进升级。哈希记录通常包含算法和参数，使登录成功时能够判断是否重新哈希。

# 工作原理

注册时算法生成盐，将密码、盐和参数送入密码哈希函数，保存编码后的哈希记录。登录时从记录读取算法参数，对输入执行相同计算并进行安全比较。

盐不需要保密；服务器级 pepper 若使用则需放在密钥管理系统，不能与数据库一起保存。普通 SHA 系列速度太快，不适合直接存储密码。

# 企业实践

- 使用成熟密码哈希库和适合当前平台的算法，不自创方案。
- 设置最大输入长度，避免恶意超长密码造成资源消耗。
- 参数基准需兼顾安全与登录容量，并定期评估。
- 修改密码需要验证身份、撤销旧会话并记录审计。
- 任何响应、异常和日志都不包含密码或哈希。
- 测试使用虚构凭证，不复制真实生产数据。

# TaskFlow Pro应用

User Model 只保存 `password_hash`，注册 Schema 接收密码但响应 Schema 永不包含哈希。认证 Service 负责哈希与验证，不让 Router 直接操作算法细节。

后续若升级工作因子，可在用户成功登录后重新哈希并更新。密码修改会提升会话版本，使 Redis/Token 校验拒绝旧会话。

# 常见误区

- 用 MD5、SHA-256 直接哈希密码。
- 所有用户共用固定盐。
- 认为盐必须像密钥一样保密。
- 把哈希算法写进多个路由，无法统一升级。
- 在测试失败输出中打印原始密码。

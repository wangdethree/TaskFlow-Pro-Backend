> 主题：FastAPI、ASGI、Uvicorn、请求生命周期与 OpenAPI。

# 背景

传统同步 Web 模型在大量 I/O 等待场景下需要更多线程或进程。ASGI 为 Python 提供异步服务器与应用的标准边界，FastAPI 在其上结合类型标注、数据校验和依赖注入，降低 API 契约与实现分离造成的重复。

企业采用框架不是因为“接口写得少”，而是需要一致的请求处理、资源生命周期、文档和扩展机制。

# 核心思想

FastAPI 把函数签名视为可执行契约：路由参数、Pydantic 模型和依赖声明共同描述如何解析请求。控制权由框架掌握，应用提供路由与依赖。

ASGI 解耦服务器和应用。Uvicorn 负责监听网络、协议解析和调用 ASGI 应用；FastAPI 负责路由、依赖、校验和响应。异步提升 I/O 等待期间的并发利用率，不会让 CPU 计算自动变快。

# 工作原理

Uvicorn 将 HTTP 事件转换为 ASGI `scope`、`receive`、`send` 调用。FastAPI/Starlette 中间件处理请求，路由匹配目标端点，框架解析依赖图和参数，Pydantic 校验数据，再执行端点并序列化响应。

同步端点通常由框架放入线程池，异步端点运行在事件循环。若在 `async def` 中直接执行阻塞数据库驱动或文件操作，会阻塞整个事件循环。

应用 lifespan 管理启动和关闭资源，例如连接池或客户端。OpenAPI Schema 根据路由、参数、Schema 和响应声明生成，Swagger UI 只是该契约的一种交互展示。

# 企业实践

- 应用工厂或集中装配负责路由、中间件和生命周期。
- 异步路径只调用非阻塞组件；CPU 重任务交给任务队列或专用执行资源。
- 启动阶段验证关键配置，但避免每个 Worker 重复执行不可幂等迁移。
- 明确代理、超时、请求体大小和 Worker 数量，而非只使用开发启动参数。
- 把 OpenAPI 当作需维护的外部契约，响应模型和错误语义不能随意漂移。
- 中间件用于跨请求关注点，不承载具体业务规则。

# TaskFlow Pro应用

`backend/app/main.py` 负责创建应用、注册版本化路由和生命周期。后续数据库、Redis、MinIO 客户端在基础设施层创建，通过依赖或应用状态访问，并在关闭阶段释放。

任务、项目 API 使用异步数据库访问；邮件、统计和 AI 分析进入 Celery，避免占用请求生命周期。OpenAPI 用于后端自测和未来展示，但不能替代自动化测试。

# 常见误区

- 把 FastAPI 和 Uvicorn 当作同一个组件。
- 认为函数写成 `async def` 就获得性能提升。
- 在启动钩子中执行每个实例都可能重复的破坏性操作。
- 用中间件查询全部业务权限。
- 只要 Swagger 能请求成功就认为接口可靠。

4. lifespan 适合管理什么，不适合执行什么？
5. OpenAPI 为什么既是文档也是契约？

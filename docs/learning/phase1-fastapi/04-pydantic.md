> 主题：Pydantic Schema、请求校验、响应模型与数据边界。

# 背景

HTTP 输入来自不可信客户端，类型、格式和字段都可能错误。若业务层直接使用原始字典，每个用例都要重复判断；响应若直接暴露 ORM Model，还可能泄露密码哈希和内部字段。

Pydantic 用声明式模型建立系统边界，在数据进入业务层前完成结构校验，在返回客户端前限制输出形状。

# 核心思想

Schema 是数据契约，不是数据库表的复制品。请求模型表达调用者允许提供什么；响应模型表达系统承诺返回什么；内部命令或结果可以根据业务需要独立建模。

验证只能保证数据形状和局部约束，不能代替需要数据库与权限上下文的业务校验。例如日期格式可以由 Schema 验证，负责人是否属于项目必须由 Service 判断。

# 工作原理

FastAPI 根据函数签名将路径、查询和 Body 数据交给 Pydantic。Pydantic Core 依据字段注解执行解析、转换和约束检查，生成模型实例或结构化错误。

序列化阶段响应模型会再次约束输出。Pydantic v2 使用明确的验证器、序列化配置和从对象属性读取等机制；具体行为应以项目锁定版本为准。

严格转换与宽松转换各有取舍。过度宽松可能把错误输入静默变成其他值，过度严格会降低客户端兼容性，应根据边界风险决定。

# 企业实践

- Create、Update、Read Schema 按语义拆分，不让用户提交服务端字段。
- Patch 更新区分“未提供”和“显式为空”，避免覆盖错误。
- 响应模型采用白名单输出，敏感字段永不进入外部 Schema。
- 复用稳定的值对象和分页结构，避免继承层次过深。
- 对跨字段的纯数据规则使用模型验证器，数据库规则留在 Service。
- 错误信息对客户端可定位，同时不暴露内部实现。

# TaskFlow Pro应用

注册请求只允许邮箱、用户名和密码，不接收 `is_admin`；用户响应不含密码哈希。任务创建 Schema 接收标题、描述、负责人和截止日期，任务响应增加数据库生成的 ID、状态和时间。

项目部分更新必须区分未传字段与传入 `null`。Agent Tool 也使用结构化 Schema，但其输入仍需经过 Service 权限和业务验证。

# 常见误区

- 为每张表只创建一个通用 Schema。
- 把数据库是否存在、用户是否有权等校验全部塞进验证器。
- 直接返回 ORM 对象且不声明响应模型。
- 用默认值掩盖客户端漏传的重要字段。
- 认为通过类型校验的数据必然符合业务规则。

# 学习检查问题

1. Schema 与 ORM Model 为什么不应一一等同？
2. 字段校验和业务校验的边界如何判断？
3. Patch 中“未提供”和 `null` 有何不同？
4. 响应模型如何降低敏感字段泄露风险？
5. Agent Tool 已有 Schema 后为什么仍需 Service 校验？

> 主题：Dependency Injection、控制反转、Depends 与请求级资源。

# 背景

如果每个路由自行创建数据库 Session、解析 Token 和构造 Service，资源释放逻辑会重复，测试也难以替换协作者。依赖注入把“需要什么”与“怎样创建”分离。

企业后端借此统一身份解析、Session 生命周期和基础设施访问，并在测试中替换外部依赖。

# 核心思想

控制反转表示对象不再主动掌控所有协作者的创建，框架或装配层负责提供。依赖注入是实现方式：函数声明需要某项能力，外部容器在调用前解析并传入。

FastAPI 的 DI 主要是函数图解析，不是通用全局服务定位器。依赖应表达请求处理所需能力，而不是把任意业务流程隐藏起来。

# 工作原理

FastAPI 启动时分析端点及 Depends 的函数签名，构建嵌套依赖图。请求到来后按拓扑关系解析参数和子依赖；同一请求中相同依赖默认可缓存结果。

带 `yield` 的依赖在 yield 前获取资源，把值交给下游，在响应处理完成或异常后执行清理逻辑，适合管理 AsyncSession 等请求级资源。

典型认证链为：提取 Bearer Token → 验证签名与声明 → 查询当前用户 → 检查账户状态 → 将用户传给端点。对象级权限还需要资源标识和业务上下文。

# 企业实践

- Session 一请求一实例，确保最终关闭；事务是否提交由业务用例决定。
- 当前用户依赖只建立可信身份，不把所有项目权限一次性查询完。
- 依赖保持小而可组合，避免隐式执行大量业务写入。
- 测试使用 dependency override 替换数据库或身份依赖，并在测试后恢复。
- 区分 Singleton 客户端、请求级 Session 和瞬时业务对象生命周期。
- 依赖异常使用稳定认证/授权错误，不泄露 Token 细节。

# TaskFlow Pro应用

`app/api/dependencies` 可提供 `get_db_session`、`get_current_user` 等入口。Router 接收注入后的当前用户与 Session，再调用 Service；项目成员权限由 Service 或专门策略结合 `project_id` 判断。

未来 Agent Tool 不使用 FastAPI Depends，但会通过独立装配获得同一 Service 和授权上下文。这说明业务规则不能只存在于 Web 依赖中。

# 常见误区

- 在全局变量中共享 AsyncSession。
- `yield` 后忘记释放资源或吞掉异常。
- 依赖函数同时完成认证、业务更新和通知发送。
- 每个端点重复解析 Token，不形成依赖链。
- 认为通过 `get_current_user` 就完成所有授权。

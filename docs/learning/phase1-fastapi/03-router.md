> 主题：APIRouter、路由模块化、`include_router` 与 API 版本设计。

# 背景

接口增加后，全部写在 `main.py` 会造成路由冲突、权限依赖重复和多人修改冲突。APIRouter 让相关端点按业务边界聚合，再由应用统一装配。

模块化路由还为版本前缀、标签、公共依赖和统一响应约定提供组合点，使入口结构可预测。

# 核心思想

Router 是一组路由声明，不是业务模块的全部实现。它把 HTTP 适配集中起来：解析请求、调用 Service、将结果转为响应。

`include_router` 是组合机制。应用根节点通过显式装配决定哪些模块对外暴露，业务 Router 可以拥有局部前缀、标签和依赖。

# 工作原理

装饰器创建路由对象并记录路径、方法、端点、响应模型和依赖。`include_router` 将子 Router 的路由复制或组合到主路由表，并叠加前缀、依赖等配置。

请求进入后，Starlette 按路由表匹配路径和方法。静态路径与动态参数可能冲突，因此声明与路径设计需要避免模糊匹配。

版本前缀如 `/api/v1` 表达外部契约世代。内部 Python 包版本不必与 URL 版本完全同构；只有不兼容外部变化才需要评估新版本。

# 企业实践

- 按用户、项目、任务等业务资源拆分 Router。
- 在版本聚合 Router 统一包含各资源，而不是在 `main.py` 导入所有端点细节。
- 公共认证依赖可放在需要保护的 Router，但对象级权限仍由具体用例判断。
- 路由函数保持薄，避免直接拼复杂 SQL 或提交事务。
- 路径顺序和命名需要测试，尤其是 `/me` 与 `/{user_id}` 等组合。
- 版本升级提供迁移期，避免同时维护大量复制代码。

# TaskFlow Pro应用

`app/api/v1/` 将聚合认证、项目、任务、评论、文件和通知 Router。`main.py` 只包含总入口，Service 不依赖 Router，因此未来 Agent Tool 可绕过 HTTP 适配但不会绕过业务层。

项目级成员校验不能只放在 Router 公共依赖中，因为不同端点需要不同资源上下文；Router 负责提取路径参数，Service 或权限策略完成对象级判断。

# 常见误区

- 一个文件一个端点，过度碎片化。
- 把 Router 当 Service，在路由里实现全部规则。
- 用 URL 版本掩盖普通字段扩展。
- 在多个位置重复 prefix，生成意外路径。
- 公共认证依赖通过后就不再检查项目数据权限。

2. `include_router` 对子路由做了什么组合？

4. 哪些变化值得创建 `/v2`，哪些可以兼容演进？
5. 为什么 Router 级登录校验不能替代对象权限？

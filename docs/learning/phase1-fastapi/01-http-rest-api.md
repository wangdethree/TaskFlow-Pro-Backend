> 主题：HTTP 请求响应、状态码、REST 资源设计与幂等性。

# 背景

前后端、脚本和第三方系统需要通过稳定协议调用后端。若每个接口自行定义语义，同一种失败可能返回不同状态，客户端只能依赖文案猜测结果，重试还可能重复创建数据。

HTTP 提供统一消息语义，REST 提供围绕资源组织接口的设计方式。企业 API 的目标不是路径看起来整齐，而是让调用者能预测行为、缓存、重试和失败处理。

# 核心思想

HTTP 是无状态的应用层协议：每个请求应携带完成处理所需的上下文，服务器通过方法、目标、头和消息体理解意图。REST 强调资源标识、统一接口和无状态交互。

方法语义比动词命名更重要。安全方法不应改变业务状态；幂等操作执行一次与重复执行的预期最终状态相同。幂等不代表响应内容完全相同，也不代表绝对没有日志等副作用。

# 工作原理

请求包含方法、路径、查询参数、Headers 和可选 Body；响应包含状态码、Headers 和可选 Body。代理、网关、框架和客户端都依赖标准语义做路由、缓存、认证和重试。

状态码按类别表达结果：2xx 成功，3xx 重定向，4xx 表示请求或权限问题，5xx 表示服务器未按契约完成。业务错误应选最接近语义的状态并提供稳定错误标识，而不是所有结果都返回 200。

GET、PUT、DELETE按语义应幂等；POST通常不保证。对创建任务等可能因网络超时被重试的操作，可引入幂等键、业务唯一约束和结果复用。

# 企业实践

- 使用名词资源路径，关系通过层级或查询条件表达。
- 分页、过滤和排序形成一致规范，避免无限列表。
- 区分认证失败、权限不足、资源不存在、状态冲突和校验错误。
- 对外错误不泄露堆栈或数据库细节，内部通过 Request ID 关联日志。
- 版本策略以兼容性为中心，避免每次字段新增都升级大版本。
- 对重试风险高的写操作明确幂等方案和过期策略。

# TaskFlow Pro应用

项目、成员、任务、评论和通知都作为资源建模。例如任务状态变化不是任意更新字段，而是受业务状态机约束；非法转换更接近资源状态冲突，而非服务器异常。

任务列表使用查询参数过滤项目、负责人和状态。创建项目、邀请成员等接口需要考虑客户端超时重试，数据库唯一约束是幂等保护的一部分，但还要定义重复请求返回行为。

# 常见误区

- 所有失败都返回 200，再在 Body 中放自定义成功标志。
- 路径充满 `get`、`create` 等动作，却没有资源模型。
- 认为 DELETE 必须每次返回完全相同 Body 才算幂等。
- 用 500 表示可预期的业务冲突。
- 只靠前端防止重复点击，不设计服务端幂等。

# 学习检查问题

1. HTTP 无状态与用户登录状态是否矛盾？
2. 安全方法和幂等方法有什么区别？
3. 非法任务状态转换为什么不应返回 500？
4. 创建任务因超时重试时，服务端怎样避免重复数据？
5. 资源不存在与当前用户无权访问时，状态设计需要考虑什么安全取舍？

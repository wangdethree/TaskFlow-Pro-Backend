> 主题：pip、requirements、版本锁定与可复现环境。

# 背景

后端应用同时依赖 Web 框架、ORM、驱动、缓存和测试工具，每个直接依赖又会引入间接依赖。仅记录“安装 FastAPI”不能保证团队、CI 和生产解析到同一版本组合。

一次安装成功只证明当时找到了可安装候选，不证明未来重建结果相同、组合经过业务测试或不存在安全问题。依赖管理要覆盖声明、解析、锁定、同步、更新和审计的完整生命周期。

# 核心思想

直接依赖表达项目主动选择的能力，间接依赖由依赖图引入。高层声明要说明“我们需要什么”，锁定结果要说明“验证过的完整环境是什么”。

TaskFlow 是最终部署的应用，比通用库更强调可重复构建，应固定 Python 版本并锁定完整依赖图。一个项目只能有一个权威依赖工作流，避免 pip、Poetry 和 uv 各自维护不同真相。

# 工作原理

pip 读取版本约束，从包索引获取元数据，递归计算兼容候选，下载 Wheel 或源码包，必要时构建，然后把包和 `.dist-info` 元数据安装到当前 `site-packages`。

解析器需要满足依赖图全部约束；若候选导致冲突会回溯，约束没有交集则失败。平台、Python 版本和可用 Wheel 也会影响结果。

`requirements.txt` 是 pip 输入格式，不天然等于锁文件。宽松约束会在每次安装重新解析；简单冻结本地环境又可能记录污染包。Poetry 以 `pyproject.toml` 和 `poetry.lock` 提供一体化工作流；uv 也支持项目声明、快速解析、锁定和同步。工具不同，治理目标相同。

# 企业实践

- 区分生产依赖、测试依赖和工程工具。
- 提交直接依赖声明与锁定结果，不提交虚拟环境。
- 在干净环境和 CI 中根据锁定结果同步并运行测试。
- 依赖升级使用独立 Pull Request，查看变更说明和依赖差异。
- 定期安全审计；可重复不等于安全，锁定也不能意味着永不更新。
- 谨慎配置公共与私有包源，防范拼写投毒和依赖混淆。

选择 pip requirements、Poetry 或 uv 时，应比较团队熟悉度、CI/Docker 集成、锁定语义和私有源需求，而不是只看流行度或速度。

# TaskFlow Pro应用

直接依赖将逐步包含 FastAPI、Uvicorn、Pydantic、SQLAlchemy、Alembic、异步 MySQL 驱动、Redis 客户端、Celery、MinIO SDK 和 pytest。每个新增包都必须说明用途、运行环境、兼容性和测试方式。

MySQL 驱动不能因为名称带 `async` 就直接采用；需验证 Python 3.11、SQLAlchemy AsyncEngine、目标平台、事务行为和维护状态。最终 README 应只提供一种权威安装流程。

# 常见误区

- 只写无上限的 `>=`，假设未来版本永远兼容。
- 把当前环境全部 freeze 后，不区分直接与间接依赖。
- 不提交应用锁文件，让每次构建重新求解。
- 混用多个管理器而不定义真相来源。
- 解析冲突时强制安装，把明确错误变成运行期故障。

# 学习检查问题

1. 声明、解析、锁定和同步分别解决什么问题？
2. 为什么 `requirements.txt` 不天然等于锁文件？
3. “能安装成功”还缺少哪些稳定性证据？
4. 直接依赖和间接依赖应怎样分别表达与管理？
5. 新增 Agent SDK 前应评估哪些兼容性与供应链风险？

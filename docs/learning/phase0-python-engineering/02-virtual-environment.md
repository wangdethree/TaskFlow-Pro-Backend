> 主题：Python 虚拟环境、`venv` 工作机制与企业环境隔离。

# 背景

不同项目常要求同一第三方包的不同版本。全部安装到系统 Python 后，一个项目的升级会改变另一个项目的运行条件；本机残留包还会掩盖依赖声明遗漏，导致代码在 CI 或服务器失败。

系统 Python 可能由操作系统或包管理器维护，随意全局安装也会造成权限、安全和升级问题。企业因此要求每个项目拥有可丢弃、可重建的独立 Python 环境。

# 核心思想

虚拟环境通过“一项目一运行前缀”隔离解释器入口和第三方包目录。它不复制完整操作系统，而是在项目范围内建立独立 `site-packages`。

环境隔离与依赖锁定是两个问题：虚拟环境决定包安装在哪里；依赖管理决定安装哪些版本。两者必须同时存在才能实现可靠重建。

# 工作原理

`venv` 基于某个基础解释器创建环境，生成 `pyvenv.cfg`、环境内 Python 入口、脚本目录和独立 `site-packages`。运行时 `sys.prefix` 指向虚拟环境，`sys.base_prefix` 指向基础解释器。

激活环境主要是把环境的可执行目录放到当前 Shell 的 `PATH` 前部，使 `python`、`pip` 等命令优先解析到该环境。激活不是启动后台服务；自动化工具也可直接指定环境解释器。

虚拟环境可能含绝对路径、平台二进制和入口脚本，因此不适合复制到其他机器。正确做法是用相同 Python 版本和锁定依赖重新创建。

# 企业实践

- 固定项目使用的 Python 主、次版本，并让本地、CI、镜像保持一致。
- 在仓库根目录使用 `.venv`，将其加入 `.gitignore`。
- 通过当前解释器调用安装工具，避免 `python` 与 `pip` 指向不同环境。
- IDE、pytest 和迁移工具显式选择项目解释器。
- 定期在干净环境验证重建流程，环境异常时优先重建。
- 虚拟环境负责 Python 包隔离，Docker 进一步固定系统库和运行用户空间。

# TaskFlow Pro应用

TaskFlow Pro 使用 Python 3.11 的独立 `.venv`。FastAPI、SQLAlchemy、pytest 等 Python 包进入该环境；MySQL、Redis、MinIO 后续可由 Docker Compose 提供。

Web 进程和 Celery Worker虽然是不同进程，但必须使用同一套经过锁定和测试的应用依赖，避免同一业务代码因环境不同产生行为分叉。

# 常见误区

- 将 `.venv` 提交 Git 或压缩发送给同事。
- 使用管理员权限向系统 Python 安装项目依赖。
- 只看命令提示符判断环境，不检查真实解释器路径。
- 多个仓库共用一个“后端环境”。
- 认为虚拟环境已经解决操作系统库和生产一致性。

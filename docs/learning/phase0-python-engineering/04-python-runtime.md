> 主题：Python 解释器、import、模块搜索路径与 `PYTHONPATH`。

# 背景

许多“本地能运行、换目录就报错”的问题并非业务缺陷，而是开发者不了解解释器入口、当前工作目录和模块搜索路径。随意在代码里修改 `sys.path` 或设置全局 `PYTHONPATH`，会让问题暂时消失，却制造不可复现的运行条件。

企业项目需要统一启动方式，并能解释代码由哪个解释器执行、模块从哪里导入、为什么不同进程看到的环境可能不同。

# 核心思想

Python 运行环境由解释器、运行前缀、搜索路径、已加载模块缓存和进程环境共同决定。导入的是“可解析的模块名”，不是编辑器中可见的任意文件路径。

稳定项目应通过正确包结构、安装方式和模块入口建立导入上下文，而不是依赖个人 Shell 的偶然路径。

# 工作原理

解释器启动后初始化 `sys.path`，其中通常包含启动入口相关目录、标准库、虚拟环境 `site-packages` 以及显式配置路径。导入时先检查 `sys.modules`，未命中则由 finder 查找模块规格、loader 创建并执行模块。

模块顶层代码在首次导入时执行。导入过程中模块会较早放入缓存，因此相互导入可能看到“部分初始化”对象，形成循环导入错误。

直接运行包内部文件会把该文件所在目录当作重要搜索起点，并可能丢失包上下文；使用 `python -m package.module` 则先按模块名定位，并保留正确的 `__package__` 语义。

`PYTHONPATH` 会向搜索路径注入目录，适合少数受控场景，但全局配置会隐藏项目未正确安装或启动的问题。不同 Shell、IDE、测试器和容器也可能读取不同值。

# 企业实践

- 固定从仓库或后端根目录使用统一模块入口启动。
- 在日志和排错中确认 `sys.executable`、工作目录和目标模块的 `__file__`。
- 避免同名文件遮蔽标准库或第三方包，例如项目文件命名为框架包名。
- 模块顶层只做确定、轻量的定义，外部连接放入生命周期管理。
- 不在业务代码中追加 `sys.path`；确需路径配置时由构建或运行系统显式管理。
- 通过包与依赖管理让测试、Uvicorn、Alembic 和 Celery 使用同一导入模型。

# TaskFlow Pro应用

Uvicorn 会根据导入字符串加载 `backend/app/main.py` 中的应用对象。pytest、Alembic 和 Celery 也需要可靠导入 `app` 包；因此项目必须统一工作目录和入口，不能每个工具各自修改路径。

未来排查“Uvicorn 能启动但 pytest 无法导入”时，应比较两者解释器、当前目录、`sys.path` 和应用包安装方式，而不是复制文件或添加随机相对路径。

# 常见误区

- 把当前工作目录等同于项目根目录。
- 用全局 `PYTHONPATH` 掩盖包结构错误。
- 认为 import 每次都会重新执行模块。
- 在模块顶层建立数据库连接或读取远程资源。
- 用局部导入隐藏所有循环依赖，而不检查架构边界。

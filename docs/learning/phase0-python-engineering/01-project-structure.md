> 主题：Python 项目目录、Package/Module、分层与依赖方向。

# 背景

一次性脚本只需要解决“现在能运行”，企业后端则需要多人长期修改、测试和部署。把路由、SQL、权限和业务判断写在同一文件后，任何改动都可能影响不相关功能，代码也无法被 Celery 或 Agent Tool 复用。

项目结构由此产生：它用目录、包和模块表达职责边界，让开发者能预测代码位置、依赖方向和修改影响。结构的价值不是美观，而是降低持续增长的认知成本。

# 核心思想

合理结构追求高内聚和低耦合。高内聚要求同一业务原因引起变化的代码靠近；低耦合要求模块只依赖协作者的稳定能力，不了解内部实现。

分层进一步按变化原因隔离协议、业务和持久化：API 理解 HTTP，Service 理解业务用例，Repository 理解数据访问。依赖通常自上而下，底层不应反向依赖路由。

分层不是每个操作都套一层空壳。只有当抽象能表达规则、控制事务、支持复用或改善测试时才有价值。

# 工作原理

一个 `.py` 文件通常是 Module，拥有独立命名空间；包含 `__init__.py` 的目录通常构成普通 Package，提供层级化导入路径。首次导入时，解释器创建模块对象、执行顶层代码并缓存到 `sys.modules`。

`__init__.py` 能标识普通包和选择性暴露公共接口，但不宜承载数据库连接或大量重新导出，否则会产生导入副作用和循环依赖。

典型调用链为：请求进入 API，Schema 校验边界数据，Service 执行业务规则和用例编排，Repository 使用 ORM 访问数据库。Model 表达持久化结构，Core 提供配置、安全和基础设施。

# 企业实践

- 使用稳定的顶级应用包和统一启动入口，避免手工修改 `PYTHONPATH`。
- 保持 `main.py` 只负责应用装配，不堆积业务逻辑。
- ORM Model 与外部 Schema 分离，避免数据库字段直接泄露为 API 契约。
- 让事务由能看到完整业务用例的层控制，而不是每个 Repository 自行提交。
- 对循环导入先检查职责和依赖方向，局部导入只能作为有依据的选择。
- `utils` 只存放真正通用的纯辅助能力，不能成为无法归类代码的垃圾桶。

按技术层组织适合建立统一边界；业务规模扩大后，可以在用户、项目、任务等模块内部继续分层。选择应服务于修改聚合，而不是追求固定模板。

# TaskFlow Pro应用

`backend/app/api/` 接收请求，`schemas/` 定义数据边界，`services/` 承载注册、成员管理和状态流转，`repositories/` 封装查询，`models/` 映射 MySQL，`core/` 提供配置和安全，`tasks/` 承载 Celery 任务。

创建任务的核心规则必须位于 Service，使 HTTP API、后台任务和未来 Agent Tool 都能复用同一套权限与状态规则。Agent 的路径固定为 `Agent → Tool → Service → Repository → Database`。

# 常见误区

- 目录越多越企业级；实际上空转层会增加跳转成本。
- 所有代码先放 `main.py`，以后再整理；边界缺失会快速形成重构债务。
- Repository 负责权限；数据访问层不应决定用户能否执行用例。
- 为缩短导入路径，在 `__init__.py` 重新导出所有对象。
- 把所有复用代码都放进 `utils`，丢失业务归属。

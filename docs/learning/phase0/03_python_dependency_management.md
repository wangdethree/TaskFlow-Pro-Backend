# Python 依赖管理

## 1. 背景

### 1.1 从“安装一个包”到“管理完整依赖图”

一个后端项目很少只依赖 Python 标准库。TaskFlow Pro 后续会使用 FastAPI、SQLAlchemy、MySQL 异步驱动、Redis 客户端、Celery、MinIO SDK 和 pytest。每个直接依赖还会继续依赖其他包，最终形成一张依赖图。

例如，项目虽然只主动选择了 Web 框架，但框架可能依赖数据校验、类型处理或 ASGI 工具。安装工具需要同时找到一组彼此兼容的版本，才能构成可运行环境。

如果团队仅通过口头说明“安装 FastAPI 和 SQLAlchemy”，会出现以下问题：

- 不同开发者安装到不同版本；
- 某个间接依赖发布新版后，新的环境突然失败；
- 本地环境长期没有重建，隐藏了缺失的依赖声明；
- CI、开发和生产环境解析出不同依赖组合；
- 为修复安全问题升级一个包，却意外破坏另一个包；
- 无法回答某个包为什么存在、由谁引入。

因此，依赖管理不是“把包安装成功”，而是管理依赖的声明、解析、锁定、安装、升级和审计整个生命周期。

### 1.2 “能安装成功”为什么不等于“环境稳定”

某次安装成功只能证明：在当时的 Python 版本、操作系统、包索引状态和版本约束下，解析器找到了一组可以安装的候选版本。

它不能证明：

- 明天重新安装仍会得到同一组版本；
- CI 和生产会得到相同结果；
- 所有依赖组合经过项目测试；
- 依赖不存在已知安全问题；
- 平台上存在兼容的二进制构建产物；
- 未来升级间接依赖不会改变行为。

稳定环境的关键是 **可重复构建、经过验证、受控更新**，而不是一次安装命令没有报错。

### 1.3 企业为什么重视依赖治理

第三方依赖属于项目供应链的一部分。企业需要知道：

- 使用了哪些直接和间接依赖；
- 每个环境实际安装了什么版本；
- 依赖升级是否经过自动化测试；
- 是否存在许可证或安全风险；
- 出现线上问题时能否重建相同环境；
- 是否能以小范围、可回滚的方式更新。

依赖管理会直接影响交付稳定性、安全性和问题排查效率。

## 2. 核心思想

### 2.1 声明、解析、锁定与同步

依赖管理可以拆成四个不同动作：

1. **声明**：项目明确自己直接需要哪些包，以及允许的版本范围。
2. **解析**：工具计算直接依赖和间接依赖的兼容版本组合。
3. **锁定**：记录已经解析并验证的具体版本及相关元数据。
4. **同步**：让当前虚拟环境与声明或锁定结果保持一致。

把这四件事区分开非常重要。仅有一份宽松声明，每次安装仍可能重新解析出不同结果；仅有当前环境中的包，也无法说明它们为何存在或如何重建。

### 2.2 直接依赖与间接依赖

**直接依赖**是项目主动使用并选择的包，例如 FastAPI、SQLAlchemy 和 pytest。

**间接依赖**是直接依赖为了工作而引入的包。项目代码可能没有主动导入它们，但它们的版本仍会影响运行行为。

企业项目需要同时管理两种信息：

- 高层声明应清楚表达项目主动选择了什么；
- 锁定结果应记录完整依赖图的可重复版本。

如果把所有间接依赖都当作直接依赖手工维护，会失去“为什么需要这个包”的信息；如果完全不锁定间接依赖，新的间接版本又可能让环境漂移。

### 2.3 应用与库的版本策略不同

TaskFlow Pro 是最终部署的应用，不是提供给其他项目复用的通用库。应用更重视部署环境可重现，因此通常会锁定完整依赖集合。

可复用库则需要声明与多个合理版本兼容，不能强迫所有使用者采用完全相同的间接版本。理解项目类型，有助于选择恰当的版本约束和锁定策略。

### 2.4 单一依赖管理入口

同一项目不应同时让 pip、Poetry 和 uv 各自维护一套互相独立的真相来源。可以比较或迁移工具，但稳定阶段要明确：

- 哪个文件表达直接依赖；
- 哪个文件记录锁定结果；
- 开发者和 CI 使用哪个命令同步；
- 依赖更新通过什么流程进入主分支。

工具不是目标。目标是让环境可解释、可重建和可验证。

## 3. 工作原理

### 3.1 pip 的角色

pip 是 Python 常用的包安装工具。它接收包名、版本约束或 requirements 文件，然后完成候选查找、依赖解析、构建和安装。

概括来说，pip 的工作过程包括：

1. 读取用户给出的需求和版本约束；
2. 从配置的包索引或其他来源获取包元数据；
3. 递归读取候选包的依赖要求；
4. 计算满足全部约束的版本组合；
5. 获取适合当前 Python 和平台的分发产物；
6. 必要时构建包；
7. 将包内容和元数据安装到当前环境；
8. 记录已安装发行包的版本、文件等信息。

pip 是安装器和解析器，但它并不天然知道项目的业务兼容性。两个包在元数据上允许共存，不代表它们组合后一定满足 TaskFlow Pro 的功能需求，最终仍需要测试验证。

### 3.2 Wheel、源码包与安装过程

Python 包常见分发形式包括：

- **Wheel**：已经构建的分发格式，安装通常更快；
- **Source Distribution（sdist）**：源码分发，安装时可能需要在本地构建。

如果目标平台存在匹配的 Wheel，pip 通常优先使用。若只能从源码构建，环境可能还需要编译器、系统头文件或数据库客户端库。这也是“同一依赖在某台电脑能装，在另一平台失败”的原因之一。

现代构建通常通过 `pyproject.toml` 声明构建系统。安装源码包时，pip 可以创建隔离的构建环境，先安装构建依赖，再生成可安装产物。这里的“构建隔离”与项目虚拟环境不是同一个概念。

安装成功后，包代码和 `.dist-info` 等元数据会进入当前环境的 `site-packages`。发行包名称与 Python 导入名称不一定完全相同，因此不能只凭 `import` 名称猜测安装包名。

### 3.3 版本解析如何工作

假设项目依赖 A 和 B：

- A 要求 C 的版本处于某个范围；
- B 要求 C 的另一个范围；
- 项目自身也可能对 C 施加约束。

解析器需要寻找这些约束的交集，并选择可用候选。如果尝试的候选导致后续依赖冲突，解析器可能回溯并选择其他版本。如果不存在交集，就无法得到满足所有条件的环境。

解析复杂度来自依赖图，而不只是项目直接声明的几行内容。约束过宽会导致环境随时间漂移；约束过窄则可能阻碍安全更新，甚至让其他依赖无法找到兼容版本。

### 3.4 常见版本约束

常见约束表达的含义如下：

| 形式 | 含义 | 主要取舍 |
|---|---|---|
| `package==1.2.3` | 只允许指定版本 | 结果明确，但需要主动更新 |
| `package>=1.2` | 允许指定版本及更高版本 | 过于宽松时可能引入不兼容升级 |
| `package>=1.2,<2.0` | 在明确范围内选择 | 表达兼容窗口，仍需锁定实际结果 |
| `package~=1.2` | 兼容版本约束 | 比单纯下限更受控，需理解其精确语义 |
| `package!=1.2.4` | 排除已知问题版本 | 适合避开特定缺陷，但要记录原因 |

语义化版本通常用主版本、次版本和修订版本表达兼容意图，但并非所有 Python 包都严格遵循同样策略。版本号只能作为信号，不能替代变更日志和项目测试。

### 3.5 依赖冲突的来源

依赖冲突常见来源包括：

- 两个直接依赖要求同一间接依赖的互斥版本范围；
- 项目固定了过旧版本，阻止其他包升级；
- Python 版本不满足候选包要求；
- 操作系统或 CPU 架构没有兼容 Wheel；
- 环境中遗留包不属于当前声明；
- 安装时混用了不同索引或私有源；
- 运行期行为不兼容，但包元数据没有准确表达。

处理冲突时不应无脑强制安装。正确过程是确认冲突链路、业务兼容窗口、可升级的直接依赖和测试范围，再决定升级、降级、替换或隔离。

### 3.6 `requirements.txt`

`requirements.txt` 是 pip 可以读取的一种 requirements 文件。它可以包含包名、版本约束、环境标记、引用其他文件等信息。

需要特别理解：

> `requirements.txt` 是安装输入格式，但它本身不自动等于完整、跨平台、带解析语义的锁文件。

如果文件只列出宽松的直接依赖，重新安装时仍会再次解析间接依赖；如果通过冻结当前环境生成精确版本，它能够记录更多实际版本，但也可能混入未使用的包、平台专属包和偶然状态。

因此企业项目需要约定它扮演什么角色。例如：

- 一个文件声明直接生产依赖；
- 单独文件声明开发和测试依赖；
- 由工具生成完整锁定结果；
- 使用 constraints 文件限制解析结果。

不能只看到文件名就假设环境已经可重复。

### 3.7 Poetry

Poetry 是集项目元数据、依赖声明、解析、锁定、虚拟环境和构建发布于一体的工具。它通常以 `pyproject.toml` 表达项目和依赖，以 `poetry.lock` 保存解析结果。

它的价值在于提供较统一的依赖工作流：

- 区分主依赖与依赖组；
- 解析并锁定完整依赖图；
- 根据锁文件安装；
- 管理项目打包元数据；
- 支持受控更新。

需要注意：工具引入自己的命令和工作流，团队必须统一版本与使用方式。若项目只需要非常简单的应用依赖管理，也要评估额外工具成本，而不是因为流行就采用。

### 3.8 uv

uv 是现代 Python 项目和包管理工具，覆盖 Python 版本、虚拟环境、依赖解析、锁定、同步和工具运行等场景，并强调性能和统一工作流。项目模式通常使用 `pyproject.toml` 表达依赖，并通过 `uv.lock` 保存跨环境解析所需的锁定信息。

它的核心价值不只是“安装更快”，还包括：

- 用单一工具完成多个环境与依赖操作；
- 明确区分依赖声明和锁定结果；
- 根据锁定结果同步环境，而不只是在现有环境上不断追加安装；
- 适合在本地和 CI 中建立一致命令。

但采用 uv 仍需团队约定：工具版本、锁文件提交策略、依赖组、升级流程和 Docker 构建方式。速度不能替代治理。

### 3.9 锁文件解决什么问题

锁文件记录解析后的具体依赖集合，通常包含直接与间接依赖版本，以及用于区分 Python 版本、平台或来源的元数据。它让不同环境能够从同一解析结果出发。

锁定的价值包括：

- 避免间接依赖在不知情情况下漂移；
- 让 CI、开发和部署使用一致版本；
- 使依赖更新成为可审查的 Git 变更；
- 方便回退到已验证组合。

锁文件不是安全保证，也不是永久冻结。项目仍需主动更新依赖、查看变更、运行测试和处理安全问题。

## 4. 企业实践

### 4.1 推荐实践

#### 明确唯一真相来源

团队必须清楚直接依赖写在哪里、锁文件是什么、环境如何同步。不要让 README、requirements 文件和 `pyproject.toml` 分别维护不同版本。

#### 提交依赖声明和锁定结果

TaskFlow Pro 属于应用项目，完整运行环境需要可重复，因此依赖声明和选定工具的锁定结果都应进入 Git。虚拟环境目录不进入 Git。

#### 区分生产依赖与开发依赖

FastAPI、SQLAlchemy 和 Redis 客户端属于运行时依赖；pytest、格式化和静态检查工具通常属于开发或测试依赖。区分依赖组可以减少生产镜像体积和攻击面，也能说明每个包存在的原因。

#### 在干净环境和 CI 中验证

依赖变更不能只在已有环境中测试。已有环境可能残留未声明包。应从干净环境按照锁定结果同步，再运行完整测试。

#### 受控升级

升级流程至少应包括：

1. 明确升级原因；
2. 阅读重要版本变更和兼容性说明；
3. 更新声明和锁定结果；
4. 检查依赖差异；
5. 运行测试和必要迁移验证；
6. 通过独立 Pull Request 合并；
7. 保留可回滚版本。

#### 定期安全审计

锁定旧版本可以带来可重复性，但也可能长期保留漏洞。企业项目需要依赖漏洞检查、更新计划和风险评估，而不是“锁住以后永远不动”。

#### 谨慎选择包来源

生产依赖应来自明确、可信的索引或内部制品库。拼写错误、未知来源包和未经审查的安装脚本都可能构成供应链风险。私有源与公共源混用时要避免名称混淆和来源优先级问题。

### 4.2 常见错误

#### 只写下限，不锁定实际结果

只写 `>=` 会让未来安装选择当时可见的新版本。即使直接依赖不变，间接依赖也可能变化，导致环境漂移。

#### 对所有包手工执行冻结并视为设计完成

冻结当前环境能记录结果，但如果环境本来已被污染，就会把无关包也写入文件；同时无法清楚区分直接和间接依赖。冻结是工具动作，不等于依赖设计。

#### 不提交锁文件

对于最终部署的应用，不提交锁文件会让每次构建重新求解依赖图，增加“昨天能运行，今天不能”的风险。

#### 混用 pip、Poetry 和 uv

如果有人更新 `requirements.txt`，有人只更新 `poetry.lock`，另一些人执行 uv 同步，项目就会出现多个互相冲突的环境来源。迁移期间可以短期共存，但必须有明确计划和单一最终入口。

#### 依赖升级与业务功能混在一个巨大提交中

这样很难判断故障来自业务改动还是依赖变化，也难以单独回滚。重要依赖升级应形成边界清晰、测试充分的变更。

#### 忽略间接依赖

项目虽然没有直接导入某个包，它仍可能位于运行链路和漏洞影响范围内。排查时要查看完整依赖图，而不是只看顶层列表。

#### 看到解析失败就强制绕过

解析失败是在提示约束无法同时满足。跳过解析或强制覆盖可能把明确的安装错误变成更隐蔽的运行错误。应定位依赖链并做有依据的版本决策。

### 4.3 requirements.txt、Poetry 与 uv 的选择

| 方案 | 优点 | 需要注意 | 适合场景 |
|---|---|---|---|
| pip + requirements 文件 | 标准工具、直观、生态普遍支持 | 需要额外约定锁定、依赖分组和同步方式 | 简单项目或希望显式学习底层流程 |
| Poetry | 项目元数据、解析、锁定、依赖组和发布流程集成 | 团队需要统一 Poetry 工作流和版本 | 希望使用成熟一体化项目管理体验 |
| uv | 速度快，覆盖 Python、环境、解析、锁定和同步 | 仍需约定锁文件、依赖组和 CI/容器流程 | 希望采用现代统一且高效的工作流 |

选择标准不应该只是个人偏好，而应评估：

- 团队熟悉程度；
- CI 和 Docker 集成；
- 锁定与跨平台需求；
- 私有包索引支持；
- 升级和审计流程；
- 项目是应用还是可复用库。

一旦选择，应在 README 中给出唯一推荐流程，并通过 CI 验证锁文件没有失配。

## 5. TaskFlow Pro 应用

### 5.1 依赖分类

TaskFlow Pro 后续依赖可以按职责分类：

| 分类 | 代表技术 | 作用 |
|---|---|---|
| Web 与协议 | FastAPI、Uvicorn | API 框架与 ASGI 服务 |
| 数据模型与校验 | Pydantic v2 | 请求、响应和配置校验 |
| 持久化 | SQLAlchemy 2.0、Alembic、MySQL 异步驱动 | ORM、迁移和数据库连接 |
| 缓存与消息 | Redis 客户端、Celery | 缓存、会话控制和异步任务 |
| 文件存储 | MinIO SDK | 附件对象操作 |
| 测试 | pytest 及异步测试支持 | 单元、集成和 API 测试 |
| 工程质量 | 格式化、静态检查、类型检查工具 | 提交前与 CI 质量检查 |

表中只表达依赖职责，不在 Batch 1 提前选择具体版本或安装包。版本选择必须结合 Python 3.11、官方兼容信息、项目功能和测试结果，在后续初始化决策中完成。

### 5.2 MySQL 驱动不能随意选择

TaskFlow Pro 要使用 SQLAlchemy 2.0 异步访问 MySQL，因此 MySQL 驱动必须满足：

- 支持 Python 3.11；
- 能与 SQLAlchemy AsyncEngine 配合；
- 维护状态和文档可靠；
- 在目标操作系统和 Docker 环境中可安装；
- 行为能够通过事务与连接测试验证。

不能因为包名中带有 `async` 就直接认定适合，也不能只看安装成功。驱动选择会在数据库阶段形成明确设计记录。

### 5.3 建议的治理目标

TaskFlow Pro 的具体工具选择会在 Phase 0 初始化设计中确认，但最终依赖流程必须实现以下目标：

```text
明确 Python 3.11
→ 声明直接依赖及用途
→ 解析完整依赖图
→ 锁定经过验证的版本
→ 在 .venv 和 CI 中一致同步
→ 运行自动化测试
→ 通过 Pull Request 受控更新
```

无论最终使用 pip requirements、Poetry 还是 uv，都必须满足：

- 只有一个权威依赖入口；
- 直接依赖和开发依赖可以区分；
- 完整版本组合可以重建；
- 锁定文件进入 Git；
- CI 使用干净环境验证；
- 依赖升级与业务功能尽量分离；
- 开发者不能依靠未声明的全局包。

### 5.4 对后续阶段的影响

依赖管理会贯穿整个项目：

- Phase 1 增加 FastAPI、Uvicorn 和 Pydantic；
- Phase 2 增加 SQLAlchemy、Alembic 和 MySQL 驱动；
- Phase 3 增加密码哈希与 JWT 相关依赖；
- Phase 6 增加 Redis、Celery、MinIO、pytest 和工程工具；
- Phase 7 增加 LLM 或 Agent SDK。

每次新增依赖都需要回答：

1. 为什么标准库或已有依赖不能满足需求？
2. 这个包由谁维护，兼容性和安全情况如何？
3. 它引入了哪些间接依赖？
4. 是否属于生产依赖？
5. 如何测试它的关键行为？
6. 如果未来替换，影响哪些模块？

这样可以避免技术栈无控制膨胀。

## 6. 学习检查

学习完成后，应当能够用自己的话回答以下问题，并能结合 TaskFlow Pro 说明决策：

1. 依赖声明、解析、锁定和同步分别解决什么问题？
2. 什么是直接依赖和间接依赖？为什么两者都需要被管理？
3. pip 从接收包名到完成安装，大致经历哪些步骤？
4. Wheel 和源码分发有什么差异？为什么同一个包可能在不同平台出现不同安装结果？
5. 依赖解析器为什么需要回溯？两个包要求互斥版本时会发生什么？
6. 为什么 `package>=1.0` 不能保证未来环境稳定？
7. `requirements.txt` 为什么不天然等同于锁文件？
8. 直接冻结一个长期使用的本地环境可能把哪些问题写入依赖文件？
9. Poetry 的依赖声明和锁定思路是什么？采用它会增加哪些团队约定？
10. uv 除了安装速度之外，还试图统一哪些工作流？
11. 为什么 TaskFlow Pro 作为应用项目适合锁定完整依赖集合？
12. 锁文件为什么应该提交 Git？它又为什么不能替代安全更新和测试？
13. “能安装成功”缺少了哪些关于稳定性和业务兼容性的证据？
14. 如果 FastAPI 的升级要求新版 Pydantic，而项目另一依赖只支持旧版，你会怎样分析，而不是直接强制安装？
15. 为什么同一项目不应长期混用 pip、Poetry 和 uv 维护不同依赖结果？
16. 生产依赖和开发依赖为什么要区分？pytest 应属于哪一类？
17. TaskFlow Pro 选择异步 MySQL 驱动时，除了“支持安装”还要验证什么？
18. 如何证明本地、CI 和部署环境使用的是同一套经过验证的依赖？
19. 新增 Agent SDK 前，需要从哪些工程和安全角度评估它？
20. 依赖升级为什么最好与大型业务功能改动分离提交？

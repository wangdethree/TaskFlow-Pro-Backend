# Python 虚拟环境

## 1. 背景

### 1.1 全局安装为什么会失控

Python 程序通常依赖大量第三方包。不同项目可能使用不同版本的 FastAPI、SQLAlchemy、Pydantic 或测试工具。如果所有项目都把依赖安装到同一个全局 Python 环境，会出现几个根本问题：

- 项目 A 需要某个包的旧版本，项目 B 需要不兼容的新版本；
- 为项目 B 升级依赖后，原本正常的项目 A 突然失败；
- 开发者电脑“恰好安装过”的包掩盖了项目依赖声明的遗漏；
- CI 或服务器没有相同的全局包，代码离开本机就无法运行；
- 卸载一个项目的包时，可能破坏其他项目；
- 很难证明当前命令究竟由哪个 Python 和哪组包执行。

问题的本质不是 Python 不能同时安装多个包，而是一个安装环境内通常只能存在某个发行包的一套有效版本。例如两个项目分别依赖互不兼容的同名版本时，它们不能可靠地共享同一个 `site-packages`。

### 1.2 系统 Python 的问题

系统 Python 可能由操作系统、包管理器或其他开发工具维护。它不属于某一个项目，随意修改会带来以下风险：

- 操作系统工具或其他程序依赖其中的包；
- 系统升级可能替换解释器或安装目录；
- 普通用户可能没有全局写权限；
- 使用管理员权限安装包会扩大误操作和供应链风险；
- 团队成员的系统 Python 来源和补丁版本不一致。

因此，系统 Python 更适合作为“创建项目环境的基础解释器”，而不是承载具体项目全部依赖的公共容器。

### 1.3 企业为什么要求环境隔离

企业项目需要满足可重复开发、自动化测试和稳定部署。环境隔离让每个项目拥有独立依赖空间，从而实现：

- 项目之间互不污染；
- 开发、CI 和部署能够基于同一依赖声明重建环境；
- 升级依赖时影响范围可控；
- 出现问题时能够确认解释器和依赖来源；
- 删除项目环境不会影响其他项目。

虚拟环境解决的是“安装位置与运行上下文隔离”。它不会自动解决依赖版本锁定，也不会让开发、测试和生产天然一致；后者还需要依赖管理、配置管理和容器化等机制配合。

## 2. 核心思想

### 2.1 一项目一环境

虚拟环境的核心思想是：

> 让每个项目拥有独立的 Python 运行前缀和第三方包目录，同时复用或关联一个基础 Python 解释器。

TaskFlow Pro 不需要复制整个操作系统，也不需要启动虚拟机。虚拟环境只隔离 Python 层面的解释器入口、安装目录和命令脚本，因此创建速度快、磁盘占用较小。

“一项目一环境”通常指一个独立部署单元或仓库拥有自己的环境，而不是每个开发者机器只创建一个叫“后端开发”的公共环境。项目之间即使目前技术栈相似，也可能在未来以不同速度升级。

### 2.2 隔离的是环境，不是声明

虚拟环境回答：包安装到哪里、运行时从哪里加载。

依赖管理回答：应该安装哪些包、允许哪些版本、如何复现同一组依赖。

二者不能互相替代：

- 只有虚拟环境但没有依赖文件，别人不知道如何重建；
- 只有依赖文件但安装到全局环境，项目之间仍然会互相污染。

### 2.3 环境应当可丢弃、可重建

企业项目不会把虚拟环境本身当作交付物。正确目标是：任何开发者和 CI 都可以使用指定 Python 版本及依赖声明重新创建环境。

因此 `.venv/` 应视为本地产物：

- 不提交 Git；
- 不作为唯一依赖记录；
- 损坏时直接重建，而不是手工修补；
- 不复制到生产服务器充当部署方案。

## 3. 工作原理

### 3.1 Python 解释器与运行前缀

执行 Python 程序时，真正运行代码的是某个具体的 Python 解释器可执行文件。解释器需要确定自己的运行前缀，并据此查找标准库、平台相关文件和第三方包。

在普通环境中，`sys.prefix` 通常指向解释器的基础安装位置。在虚拟环境中：

- `sys.prefix` 指向当前虚拟环境；
- `sys.base_prefix` 指向创建它的基础 Python；
- 两者不同通常意味着当前正在虚拟环境中运行。

这个前缀变化使解释器优先使用虚拟环境自己的安装目录，而不是把项目依赖安装到基础环境。

### 3.2 `site-packages` 是什么

`site-packages` 是第三方发行包通常被安装和发现的位置。全局 Python 有自己的 `site-packages`，每个虚拟环境也有独立目录。

当 TaskFlow Pro 的虚拟环境中安装 FastAPI 时，FastAPI 及其相关元数据进入该环境的 `site-packages`。另一个项目的虚拟环境拥有不同目录，因此可以安装不同版本而不直接覆盖 TaskFlow Pro。

默认情况下，`venv` 创建的环境不会继承系统 `site-packages`。虽然可以显式开启继承，但这会削弱隔离，使项目依赖再次受到系统环境影响，企业项目通常不使用该选项。

### 3.3 `venv` 创建了什么

使用 Python 标准库的 `venv` 创建环境时，通常会生成：

- 一个环境配置文件 `pyvenv.cfg`；
- 指向或复制自基础解释器的 Python 可执行入口；
- 独立的第三方包目录；
- `pip` 等环境内命令入口；
- 用于激活环境的脚本。

具体是复制还是符号链接，取决于操作系统和创建方式。`pyvenv.cfg` 会记录基础 Python 位置等信息，解释器据此识别自己处于虚拟环境。

虚拟环境不包含独立操作系统内核，也不提供容器级文件系统或网络隔离。它只解决 Python 运行环境隔离，所以不能替代 Docker。

### 3.4 激活到底做了什么

“激活虚拟环境”主要是修改当前 Shell 会话的环境变量：

- 把虚拟环境的可执行文件目录放到 `PATH` 前面；
- 通常修改命令提示符，提示当前环境；
- 保存必要的恢复信息，以便停用时还原。

激活后输入 `python` 或 `pip`，Shell 会优先找到虚拟环境内的入口。

激活不是开启一个特殊 Python 模式，也不是后台启动服务。即使不激活，只要显式执行虚拟环境中的 Python 路径，仍然会使用该环境。CI、编辑器和进程管理工具经常直接指定解释器，而不依赖交互式激活脚本。

### 3.5 创建和使用方式

TaskFlow Pro 固定使用 Python 3.11。创建环境前，应先确认用于创建环境的解释器确实是目标版本，而不是仅凭命令名称猜测。

在项目根目录可以执行：

```bash
python3.11 -m venv .venv
```

在 macOS 或 Linux 的常见 Shell 中激活：

```bash
source .venv/bin/activate
```

在 Windows PowerShell 中激活：

```powershell
.venv\Scripts\Activate.ps1
```

完成后应检查：

```bash
python --version
python -c "import sys; print(sys.executable); print(sys.prefix); print(sys.base_prefix)"
python -m pip --version
```

推荐使用 `python -m pip`，因为它明确表示“使用当前这个 Python 解释器加载 pip 模块”，比单独执行 `pip` 更不容易误用其他环境中的命令。

退出当前 Shell 的虚拟环境：

```bash
deactivate
```

这些命令只用于学习环境机制，本阶段不会安装 TaskFlow Pro 的业务依赖或生成项目代码。

### 3.6 为什么虚拟环境通常不可移动

虚拟环境中的解释器入口和已安装命令可能包含创建时的绝对路径或平台相关信息。把 `.venv` 直接复制到另一目录、另一台机器或不同操作系统后，入口脚本和二进制扩展可能失效。

因此虚拟环境应在目标位置重新创建，再根据依赖声明安装，而不是作为压缩包在团队成员之间传递。

### 3.7 虚拟环境与 Docker 的区别

| 能力 | Python 虚拟环境 | Docker 容器 |
|---|---|---|
| Python 包隔离 | 是 | 是 |
| Python 解释器版本控制 | 依赖基础解释器 | 可由镜像固定 |
| 操作系统用户空间隔离 | 否 | 是 |
| 系统库隔离 | 否 | 是 |
| 网络和进程隔离 | 否 | 是 |
| 本地开发启动速度 | 快 | 相对更重 |
| 主要用途 | Python 开发环境 | 服务运行与部署环境 |

两者不是互斥关系。TaskFlow Pro 本地开发可以使用 `.venv` 获得快速反馈，同时用 Docker Compose 启动 MySQL、Redis 和 MinIO；部署阶段再通过容器固定完整运行环境。

## 4. 企业实践

### 4.1 推荐实践

#### 固定 Python 大版本和次版本

项目声明 Python 3.11，不应让部分成员使用 3.10、部分成员使用 3.13 后再假设行为完全相同。CI 和部署也应使用同一目标版本，并在升级时通过独立变更和完整测试验证。

#### 在项目根目录使用 `.venv`

`.venv` 命名便于编辑器自动发现，也让环境和仓库建立清晰对应关系。它必须加入 `.gitignore`。

#### 始终确认解释器来源

遇到“明明安装了却无法导入”时，应先检查：

- 当前 `python` 指向哪里；
- 当前 pip 属于哪个 Python；
- 编辑器和测试工具使用哪个解释器；
- 包是否安装到当前环境。

很多问题并不是包没有安装，而是“安装命令”和“运行命令”使用了不同解释器。

#### 通过依赖文件重建，而不是传递环境目录

开发者、CI 和服务器都应从版本化依赖声明创建新环境。这样才能验证项目没有依赖某台机器的偶然状态。

#### 自动化任务显式指定环境

IDE、测试任务、格式化工具和脚本要使用项目虚拟环境的解释器。不要假设自动化进程会读取个人 Shell 的激活状态。

#### 保持环境可丢弃

依赖异常时可以删除并重建 `.venv`。如果重建后项目不能运行，说明依赖声明、系统依赖或文档不完整，这正是需要修复的工程问题。

### 4.2 常见错误

#### 使用管理员权限全局安装项目依赖

这会污染系统环境、改变文件权限，并扩大安装恶意包时的影响范围。项目依赖应安装在虚拟环境或受控容器中。

#### 只看命令提示符判断环境

提示符只是激活脚本提供的视觉提示，可能被主题隐藏或错误修改。真正可靠的是检查 `sys.executable`、`sys.prefix` 和 pip 路径。

#### 把 `.venv` 提交到 Git

虚拟环境文件数量多、包含平台差异和本地绝对路径，既膨胀仓库，也无法可靠跨平台复用。Git 应保存重建环境所需的声明，而不是保存环境本身。

#### 多个项目共用一个环境

即使初期依赖相同，后续升级节奏和工具依赖也会分叉。共享环境把一个项目的依赖变更变成所有项目的潜在故障。

#### 混用多个安装入口

先用全局 `pip` 安装一部分包，再用虚拟环境 pip 安装另一部分，会产生“某些终端能运行、某些不能”的不确定状态。应统一通过当前解释器执行安装工具。

#### 把虚拟环境等同于生产一致性

虚拟环境不隔离系统级动态库和操作系统差异。带有二进制扩展或数据库客户端时，本地可以运行并不保证 Linux 服务器一定相同。生产一致性需要依赖锁定、CI 和容器镜像共同保证。

### 4.3 企业开发环境生命周期

一个可靠的开发环境流程通常是：

1. 安装项目指定的 Python 版本；
2. 在仓库根目录创建独立虚拟环境；
3. 根据版本化依赖声明同步包；
4. 配置本地环境变量；
5. 运行测试和健康检查；
6. 依赖变更后更新锁文件并在干净环境验证；
7. 环境损坏或来源不明时删除并重建。

CI 每次从干净环境执行，是检查“项目能否被重建”的重要手段。开发者电脑上长期存在的环境不能作为唯一正确性证据。

## 5. TaskFlow Pro 应用

### 5.1 环境边界

TaskFlow Pro Backend 将拥有自己的 `.venv`，只安装本项目开发和测试需要的 Python 包。它不会与其他 Django、FastAPI 或 Agent 项目共享环境。

项目目标版本为 Python 3.11，以下位置应保持一致：

- 开发者本地虚拟环境；
- IDE 和命令行工具；
- pytest 测试环境；
- CI 环境；
- 后续 Docker 后端镜像；
- Celery Worker 运行环境。

FastAPI Web 进程和 Celery Worker 虽然是不同进程，但它们属于同一后端代码库，并需要使用经过一致验证的项目依赖集合，避免同一业务模型在两个进程中因版本不同而表现不一致。

### 5.2 本地服务与 Python 环境

本地开发环境可以分为两类：

- `.venv` 管理 Python 解释器入口及 FastAPI、SQLAlchemy、pytest 等 Python 包；
- Docker Compose 管理 MySQL、Redis、MinIO 等外部服务。

这样既保留本地代码修改和调试的速度，也避免要求每位开发者手工安装完全相同的数据库和缓存服务。

### 5.3 TaskFlow Pro 的环境规则

- `.venv/` 必须被 Git 忽略；
- 不使用系统 Python 全局安装项目依赖；
- 不提交本地环境目录；
- 文档中的安装和测试命令默认基于项目环境；
- 使用 Python 3.11 创建环境；
- 安装工具通过当前解释器调用；
- 新成员必须能根据 README 和依赖声明从零重建；
- 依赖问题首先在干净环境复现，不能只修补个人环境；
- 后续 Docker 镜像也要使用相同 Python 主、次版本。

### 5.4 Phase 0 的应用结果

本阶段最终需要建立的不是一个“永远不变的 `.venv`”，而是一套可验证流程：

```text
指定 Python 3.11
→ 创建隔离环境
→ 根据依赖声明同步
→ 确认解释器与安装位置
→ 启动和测试项目
→ 随时可删除并重新创建
```

这个流程会成为后续数据库迁移、pytest、Celery 和 Agent 开发的共同环境基础。

## 6. 学习检查

学习完成后，应当能够用自己的话回答以下问题，并能实际检查本机环境：

1. 为什么两个当前依赖相同的项目仍然不应该长期共用一个 Python 环境？
2. 系统 Python 和项目 Python 环境的职责有什么区别？为什么不推荐管理员权限全局安装？
3. 虚拟环境主要隔离了什么？它没有隔离什么？
4. `site-packages` 在 Python 运行环境中承担什么角色？
5. `sys.prefix` 与 `sys.base_prefix` 能帮助判断什么？
6. `venv` 创建环境时大致生成了哪些内容？
7. 激活虚拟环境实际上改变了什么？不激活时是否完全无法使用该环境？
8. 为什么推荐 `python -m pip`，而不是不加判断地执行 `pip`？
9. “已经安装 FastAPI，但 import 失败”时，你会按什么顺序排查解释器和安装位置？
10. 为什么 `.venv` 不应提交 Git，也不应复制给其他开发者？
11. 虚拟环境损坏后，为什么推荐重建而不是长期手工修补？
12. 虚拟环境和 Docker 分别解决什么层面的隔离问题？
13. TaskFlow Pro 的 Web 进程和 Celery Worker 为什么应使用一致的依赖集合？
14. 如果本地 Python 是 3.11、CI 是 3.12，可能带来哪些不确定性？
15. 怎样证明一个项目环境真正可重建，而不是只在当前电脑上“碰巧能运行”？

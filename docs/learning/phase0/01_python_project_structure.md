# 企业级 Python 项目结构设计

## 1. 背景

### 1.1 从“能运行的脚本”到“可以协作的系统”

初学 Python 时，把变量、数据库查询、业务判断和输出全部写进一个文件很常见。代码规模较小时，这种方式直接、反馈快，也便于理解程序的顺序执行。

但企业后端不是一次性脚本。它会持续增加用户认证、权限、项目、任务、评论、文件和通知等功能，还会被多人修改、反复测试并长期运行。此时，如果所有代码都集中在一个文件中，会逐渐出现以下问题：

- 修改登录逻辑时，意外影响任务接口；
- 数据库代码、HTTP 代码和业务规则相互缠绕，无法单独测试；
- 多个开发者频繁修改同一个文件，产生大量合并冲突；
- 同一套权限判断被复制到多个接口，修改时容易遗漏；
- 模块导入互相依赖，最终形成循环导入；
- 新成员无法快速判断一段代码应该放在哪里；
- 程序虽然可以运行，却难以维护、替换和扩展。

因此，项目结构并不是为了让目录“看起来专业”，而是为了管理不断增长的认知复杂度。它需要让开发者看到目录和文件名时，就能大致判断代码的职责、依赖和修改影响范围。

### 1.2 企业为什么需要统一结构

企业开发具有长期维护和团队协作的特点。统一结构至少解决四类问题：

1. **定位问题**：看到接口问题时知道先检查 API、Service 还是 Repository。
2. **控制影响范围**：修改数据访问实现时，尽量不改变业务层和 API 层。
3. **支持测试**：业务逻辑可以脱离 HTTP 请求进行测试，数据查询也可以独立验证。
4. **建立协作约定**：团队对“代码应该放在哪里”形成共同语言。

结构不能消除复杂度，但能把复杂度分配到边界清晰的模块中。

## 2. 核心思想

### 2.1 分解与边界

项目结构的本质是根据变化原因拆分代码，并在模块之间建立明确边界。

例如：

- HTTP 协议细节会因为接口规范变化；
- 业务规则会因为产品需求变化；
- 数据查询会因为表结构和性能变化；
- 配置会因为部署环境变化。

这些代码变化的原因不同，就不应该全部放在同一个位置。合理拆分后，每个部分只关注自己的职责，并通过稳定接口协作。

### 2.2 高内聚与低耦合

**高内聚**是指一个模块中的内容围绕同一个职责组织。例如，任务状态流转规则集中在任务业务模块中，而不是散落在路由、数据库模型和工具函数里。

**低耦合**是指模块对其他模块的内部细节依赖较少。例如，API 层只调用任务 Service 提供的业务方法，不需要知道 SQLAlchemy 查询如何编写。

二者需要同时考虑：

- 只追求拆分，可能产生大量只有几行代码、彼此跳转频繁的文件；
- 只追求集中，可能形成职责混杂的“上帝模块”；
- 合理结构应当让相关代码靠近，同时让不同变化原因的代码分开。

判断一个拆分是否合理，可以问：

> 这组代码是否因为同一种原因而变化？修改它是否迫使不相关模块一起修改？

### 2.3 分层与依赖方向

TaskFlow Pro 采用以下基本调用方向：

```text
API Layer → Service Layer → Repository Layer → Database
```

分层的目的不是增加文件数量，而是把协议、业务和数据访问三类问题分开：

- API 层理解 HTTP；
- Service 层理解业务用例；
- Repository 层理解持久化查询；
- Database 基础设施理解数据库连接和会话。

依赖方向同样重要。Repository 不应该反向导入 API 路由，Service 也不应该依赖具体的 HTTP Request 才能执行核心业务。否则下层会被上层协议绑住，脱离 Web 环境就难以复用和测试。

### 2.4 结构服务于业务，而不是服务于形式

分层是一种管理复杂度的手段，不是规定每个请求必须机械地穿过同样数量的文件。项目早期可以保持简单，随着稳定的职责和复用需求出现再提取抽象。

一个只调用一次、没有业务含义的包装方法，未必值得单独建立一层；但权限判断、事务编排和任务状态规则具有明确业务含义，应放在 Service 层集中管理。

## 3. 工作原理

### 3.1 Python Module

Python Module（模块）通常是一个 `.py` 文件，也是组织代码和建立命名空间的基本单位。

假设存在一个名为 `task_service.py` 的文件，解释器导入它时，会为该模块创建模块对象，在模块自己的全局命名空间中执行顶层语句，并将函数、类和变量绑定到这个模块对象上。

模块带来的关键价值是命名空间隔离。两个模块都可以定义名为 `create` 的函数，只要通过各自的模块路径引用，就不会因为名称相同而直接冲突。

需要注意：模块的顶层代码通常在首次导入时执行。顶层如果发起数据库连接、读取不稳定外部资源或执行耗时操作，会让导入产生隐蔽副作用，增加测试和启动失败的风险。因此企业项目通常要求模块导入尽可能轻量和可预测。

### 3.2 Python Package

Python Package（包）是一组相关模块和子包形成的命名空间。包让项目可以使用层级化导入路径，例如 `app.services`、`app.repositories`，从而表达代码归属。

包不是简单的文件夹分类。它同时承担：

- 组织模块；
- 建立稳定导入路径；
- 定义对外暴露边界；
- 降低全局命名冲突；
- 支持测试、分发和工具发现。

传统的普通包包含 `__init__.py`。现代 Python 还支持没有 `__init__.py` 的 namespace package，但它主要解决一个逻辑包分布在多个目录或发行物中的问题。对于单仓库应用代码，显式使用普通包通常更容易理解和管理。

### 3.3 `__init__.py` 的作用

`__init__.py` 的主要作用包括：

1. 明确目录是一个普通 Python 包；
2. 在包首次导入时执行初始化语句；
3. 有选择地定义包的公开接口；
4. 保存包级元数据。

但“可以执行初始化逻辑”不意味着应该把大量逻辑放进去。常见错误是为了缩短导入路径，在多个 `__init__.py` 中重新导出大量 Model、Service 和 Repository。这样容易造成：

- 导入某个简单对象时连带加载大量模块；
- 初始化顺序变得隐蔽；
- 循环导入更难定位；
- 阅读代码时无法直接找到对象的真实定义位置。

TaskFlow Pro 初期会让 `__init__.py` 保持轻量，只有出现明确、稳定的公共接口时才选择性导出。

### 3.4 import 的执行过程

当 Python 执行导入时，可以把过程概括为：

1. 根据导入名称确定目标模块；
2. 先检查 `sys.modules` 中是否已有对应模块对象；
3. 如果没有缓存，导入系统通过 finder 查找模块规格；
4. 根据包路径和 `sys.path` 等位置找到模块；
5. 创建模块对象，并在其命名空间中执行顶层代码；
6. 将模块对象放入 `sys.modules`，再绑定到当前命名空间。

`sys.modules` 缓存意味着同一进程中模块通常只在首次导入时执行一次，但它不等于磁盘文件永远不会变化，也不能用来保存需要可靠持久化的业务状态。

`sys.path` 是模块搜索路径列表，通常受以下因素影响：

- 启动入口和当前运行方式；
- Python 标准库位置；
- 当前虚拟环境的 `site-packages`；
- 环境或工具注入的路径。

这解释了一个常见现象：直接运行包内部文件时可以出现 `ModuleNotFoundError`，而从项目根目录以模块方式运行却正常。问题通常不在“Python 找不到文件”，而在启动方式改变了包上下文和搜索路径。

企业项目不应依赖随意修改 `sys.path` 来修补导入。应使用稳定包结构、统一启动入口和一致的工作目录。

### 3.5 绝对导入、相对导入与循环导入

绝对导入从顶级包开始表达依赖，阅读时更清楚对象来自哪里；相对导入根据当前包位置查找，适合关系紧密且边界清晰的包内部代码。项目可以选择约定，但应避免混乱使用导致路径难以理解。

循环导入通常表现为：模块 A 导入模块 B，而 B 在完成初始化前又导入 A。此时 A 可能只是“部分初始化”的模块，目标名称尚未绑定，从而报错。

循环导入往往不是单纯的语法问题，而是在提示职责或依赖方向存在问题。常见改进方式包括：

- 把共享类型移动到更稳定的下层模块；
- 让依赖方向保持单向；
- 通过函数参数或依赖注入传入协作者；
- 将仅用于类型检查的导入与运行期导入区分；
- 必要时局部导入，但先确认它不是在掩盖架构问题。

### 3.6 从请求到数据的分层协作

以“创建任务”为例，合理的职责流转是：

1. API 层接收 HTTP 请求，由 Schema 校验字段格式；
2. API 层把当前用户和已校验数据交给 Service；
3. Service 判断用户是否属于项目、能否创建任务、负责人是否合法；
4. Service 在正确事务边界内调用 Repository；
5. Repository 使用 SQLAlchemy 执行持久化；
6. Service 返回业务结果，API 层转换为响应 Schema 和状态码。

如果 API 层直接完成所有查询和判断，短期代码可能更少，但业务规则会与 HTTP 紧密绑定。以后 Celery 任务或 Agent Tool 也要创建任务时，只能复制同样逻辑，最终产生不一致。因此核心业务应通过 Service 复用。

## 4. 企业实践

### 4.1 推荐实践

#### 使用唯一且稳定的应用包

应用代码应位于明确的顶级包中，例如 `app`。统一入口可以减少同名模块冲突，也让测试、迁移工具和运行命令使用一致的导入路径。

#### 保持依赖方向可解释

在 Code Review 中，不只检查代码功能，还要检查新导入是否破坏分层。例如 Repository 导入 FastAPI 的 Request 对象，通常说明数据层已经依赖 HTTP 层。

#### 按职责命名

`utils` 很容易变成无法分类代码的垃圾桶。只有真正通用、无业务归属的纯辅助能力才适合放入其中。任务状态规则应属于任务业务模块，而不是因为“多处使用”就放入 `utils`。

#### 区分 ORM Model 与 Pydantic Schema

ORM Model 表达数据库持久化结构；Schema 表达系统边界上的输入输出。直接把 ORM 对象作为所有接口模型，会让数据库字段和外部 API 强耦合，也可能意外暴露密码哈希或内部状态。

#### 通过模块入口启动应用

统一从项目根目录启动应用，使包上下文和导入路径保持稳定。不要让每位开发者通过手工添加路径解决导入问题。

#### 使用测试验证边界

Service 测试关注业务规则，Repository 集成测试关注实际数据库行为，API 测试关注协议和模块协作。边界清晰后，测试失败更容易定位。

### 4.2 常见错误

#### 所有代码写在 `main.py`

`main.py` 应主要承担应用装配：创建 FastAPI 应用、注册路由和生命周期。把模型、查询和业务逻辑全部塞进去，会形成高耦合入口文件。

#### 每一层都只是机械转发

如果 API 调用 Service，Service 原样调用 Repository，每层都没有独立职责，说明抽象可能过早。保留层次的理由应是业务边界、事务控制、复用或测试，而不是目录模板要求。

#### 在 Repository 中提交所有事务

一个业务用例可能需要写入任务、创建通知并记录审计。如果每个 Repository 方法自行 `commit`，Service 无法把多个操作作为一个整体回滚。事务通常应由能看见完整业务用例的层控制。

#### 依靠通配符导入

通配符导入会隐藏名称来源、增加冲突，也让静态分析和重构更困难。企业代码应让依赖来源清晰。

#### 为解决循环导入随意移动导入语句

局部导入有合理场景，但如果不分析依赖关系，只把导入移动到函数内部，可能只是延迟问题。应先检查模块职责是否互相穿透。

### 4.3 结构演进原则

项目结构需要随业务增长演进。开始阶段可以按层组织，业务模块增多后也可以在模块内部继续分层。无论采用哪种形式，都要保持：

- 相同业务概念尽量靠近；
- 跨模块依赖经过稳定接口；
- 基础设施不决定上层业务规则；
- 结构变化有明确收益并经过 Review。

## 5. TaskFlow Pro 应用

TaskFlow Pro 初始结构中的主要目录职责如下：

| 路径 | 职责 | TaskFlow Pro 示例 |
|---|---|---|
| `backend/app/main.py` | 应用装配与入口 | 创建 FastAPI 应用、注册路由和生命周期 |
| `backend/app/api/` | HTTP 协议层 | 认证、项目、任务、评论等路由与依赖 |
| `backend/app/core/` | 全局基础能力和策略 | 配置、安全、日志、异常基础定义 |
| `backend/app/database/` | 数据库基础设施 | Engine、Session 工厂、Base 和连接生命周期 |
| `backend/app/models/` | SQLAlchemy ORM 模型 | User、Project、ProjectMember、Task 等表映射 |
| `backend/app/schemas/` | Pydantic 输入输出模型 | 注册请求、任务创建请求、项目响应 |
| `backend/app/services/` | 业务用例与规则 | 注册用户、添加成员、流转任务状态 |
| `backend/app/repositories/` | 数据访问 | 按邮箱查询用户、分页查询任务、保存项目 |
| `backend/app/tasks/` | Celery 异步任务 | 异步通知、统计、邮件和 AI 分析 |
| `backend/app/utils/` | 少量通用辅助能力 | 与具体业务无关的纯工具；禁止成为杂物目录 |
| `backend/tests/` | 自动化测试 | 单元测试、数据库集成测试和 API 测试 |
| `backend/alembic/` | 数据库迁移 | 建表、约束、索引和后续结构演进 |

未来 Agent 的调用路径必须保持为：

```text
Agent → Tool → Service → Repository → Database
```

Agent Tool 与 API 都是系统入口，它们可以复用同一 Service。这样任务创建、项目访问和权限规则只有一个权威实现，避免 Agent 绕过安全边界。

Phase 0 不会立即创建全部业务文件。我们先理解边界，再建立最小结构；具体模块在相应阶段出现时逐步增加。

## 6. 学习检查

学习完成后，应当能够用自己的话回答以下问题，并能结合 TaskFlow Pro 给出例子：

1. 一个 `.py` 文件成为模块后，相比把代码复制到其他文件中，获得了哪些能力？
2. 模块首次导入时发生什么？为什么不应在模块顶层执行数据库连接等重操作？
3. 普通 Package 与 Module 有什么区别？`__init__.py` 解决什么问题？
4. 为什么应用项目通常保留轻量的 `__init__.py`，而不是集中重新导出所有对象？
5. Python 根据什么寻找要导入的模块？为什么从错误目录直接运行内部文件可能导入失败？
6. `sys.modules` 在导入中承担什么角色？它为什么不能当作业务持久化方案？
7. 循环导入为什么经常意味着依赖方向或职责设计存在问题？
8. 高内聚和低耦合分别是什么？请用任务状态流转举例说明。
9. 创建任务时，参数校验、权限判断、事务控制和 SQL 查询应分别由哪一层负责？为什么？
10. 为什么 ORM Model 和 Pydantic Schema 不应该被当作同一种模型使用？
11. 如果 Celery Worker 和 Agent Tool 都需要“创建任务”，业务规则应该放在哪里才能避免复制？
12. 什么情况下新增一层抽象有价值？什么情况下它只是无意义的机械转发？
13. `utils` 目录为什么容易失控？判断一段代码是否属于 `utils` 时应该问什么？
14. 如果 Repository 中每个写方法都自行提交事务，会对跨多表业务造成什么问题？
15. 面对一个越来越大的项目，你会根据哪些信号决定调整目录结构？

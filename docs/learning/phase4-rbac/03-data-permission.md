> 主题：对象级数据权限、查询范围与防水平越权。

# 背景

用户拥有“查看任务”功能权限，不代表能查看所有项目任务。攻击者常通过替换 URL 中的项目或任务 ID 访问他人数据，这属于水平越权。

数据权限决定主体能操作哪些具体记录，是企业多租户和项目协作系统的核心安全边界。

# 核心思想

授权不能先全局查询对象，再只检查通用角色。更安全的思想是把可见范围编入查询条件，使“不属于用户作用域的数据”从数据访问结果中就不存在。

同时要防止资源 ID 与父级 ID 不一致。例如 `/projects/A/tasks/T` 必须验证 T 属于 A，不能分别验证用户能看 A、系统存在 T 就返回 T。

# 工作原理

数据权限通常基于成员关联、所有权、组织层级或租户标识。查询任务时通过项目成员关系限制范围；更新时使用同时包含任务 ID、项目 ID 和授权条件的查询。

“不存在”和“无权访问”有时统一返回，以减少资源枚举；内部审计仍记录真实拒绝原因。缓存对象也必须按用户或安全作用域隔离。

# 企业实践

- 列表查询从授权范围出发，不查询全部后再在 Python 过滤。
- 所有详情、修改、删除和下载接口都做对象级校验。
- 子资源验证父子归属，防 ID 拼接越权。
- 文件下载使用短期预签名 URL 前先检查项目权限。
- 管理员跨范围查询显式开启并记录审计。
- 用不同用户、不同项目构造负向测试矩阵。

# TaskFlow Pro应用

用户只能列出自己加入项目的任务。访问评论和附件时，权限沿 `comment/file → task/project → project_member` 关系判断。

Agent 的项目分析 Tool 必须把当前用户作用域传给 Service，统计 SQL 也只能聚合授权项目，不能先读取全库后要求模型“自行忽略”。

# 常见误区

- 有某项功能权限就允许访问所有同类数据。
- 列表在 Python 内过滤，导致泄露和性能问题。
- 只验证路径中的项目，不验证任务确属该项目。
- 预签名文件 URL 永久有效或不检查权限。
- Agent 查询全库后靠 Prompt 限制输出。

# 学习检查问题

1. 功能权限与数据权限分别回答什么问题？
2. 为什么授权范围应尽量进入数据库查询？
3. 子资源父子归属如何防止 ID 拼接越权？
4. 统一返回不存在有什么安全收益和排错成本？
5. Agent 统计工具怎样确保只聚合用户可见项目？

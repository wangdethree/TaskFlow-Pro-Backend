> 主题：权限粒度、策略组合、授权入口与缓存。

# 背景

权限粒度过粗会让普通成员获得危险能力，过细则形成难以维护的权限海洋。授权逻辑散落在路由和前端按钮中，又会造成规则不一致和绕过。

企业权限设计需要稳定命名、集中判断、明确作用域和可验证矩阵。

# 核心思想

权限描述“对某类资源执行某个动作”，角色把权限组合成职责。授权决策还需要主体、资源、上下文和策略，不只是查询一个布尔值。

前端可用权限控制界面体验，但后端是权威执行点。粗粒度登录或功能校验可在依赖层完成，对象级业务权限应结合 Service 用例。

# 工作原理

授权流程通常为：认证主体 → 确定作用域 → 加载角色/权限 → 评估显式规则 → 返回允许或拒绝。规则可包含所有权、项目成员状态和目标资源状态。

权限缓存减少关联查询，但授权数据变化必须使缓存失效。可以使用短 TTL、版本号或主动删除；任何缓存故障都不能默认放行。

# 企业实践

- 使用资源与动作命名权限，保持代码稳定。
- 用权限矩阵记录 Admin、Manager、Member 的允许与拒绝。
- 关键拒绝记录安全审计，但避免记录敏感内容。
- 对批量操作逐项或按安全查询范围验证，不能只验证第一个对象。
- 缓存失效策略与角色修改处于同一设计中。
- 权限服务返回决策，错误响应避免暴露目标资源细节。

# TaskFlow Pro应用

权限可包括 `project:update`、`member:manage`、`task:create`、`task:assign`、`comment:create`。具体接口先确认项目成员，再结合角色权限和资源状态。

例如 Member 可能能编辑自己创建的评论，但不能删除其他人的评论；这需要权限与所有权策略组合，而不是新建“SelfCommentEditor”角色。

# 常见误区

- 每个特殊情况都增加一个新角色。
- 在 Router、Service 和 Repository 各写一套判断。
- 前端隐藏按钮就视为完成授权。
- 缓存读取异常时默认允许。
- 批量接口只检查调用者有某个通用权限。

# 学习检查问题

1. 权限粒度应如何避免过粗和过细？
2. 功能权限、所有权规则和资源状态如何组合？
3. 为什么 Repository 不应独立决定授权？
4. 权限缓存失败时系统应怎样处理？
5. 批量删除任务需要哪些额外授权考虑？

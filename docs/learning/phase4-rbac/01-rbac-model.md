> 主题：RBAC 用户、角色、权限模型及其边界。

# 背景

逐用户配置权限会随着人员和功能增长迅速失控，也难以审计同岗位是否拥有一致能力。RBAC 用角色承载权限，再把用户分配到角色，降低授权管理复杂度。

但简单角色判断无法表达“用户是项目 A 的 Manager、项目 B 的 Member”，TaskFlow 还需要资源作用域。

# 核心思想

RBAC 的基本关系是用户—角色—权限。权限描述原子操作，角色是业务职责集合，用户通过角色获得权限。角色不应只是代码中的 if/else 字符串。

系统角色与项目角色作用域不同：Admin 可管理系统级能力，Manager/Member 更多依附项目成员关系。最小权限要求默认拒绝，只有明确授予才允许。

# 工作原理

典型数据模型用用户角色、角色权限关联表表达多对多关系。授权时先确定主体及作用域角色，再判断角色是否包含目标权限。

若角色绑定在 `project_members`，同一用户在不同项目可拥有不同角色。权限结果可缓存，但缓存键必须包含用户、作用域和权限版本，角色变化时需要失效。

# 企业实践

- 权限使用稳定代码，如 `task:create`，显示名称可以变化。
- 角色是权限集合，不在代码里到处比较中文名称。
- 权限矩阵进入设计文档和自动化测试。
- 管理员旁路能力要显式、审计且范围有限。
- 角色变更遵守最小权限和职责分离。
- 避免为每个细微差异创建大量角色导致角色爆炸。

# TaskFlow Pro应用

Admin 负责系统管理；项目创建者或 Manager 管理成员和任务；Member 执行被允许的协作操作。项目角色保存在成员关系中，不是 User 的单一全局字段。

Service 根据权限代码和项目作用域判断用例，Agent Tool 通过同一授权服务，不能声明“AI 是管理员”绕过模型。

# 常见误区

- 在 User 表只放一个全局 role 字段。
- 角色名称直接等于权限，不建立权限集合。
- 所有管理员操作无条件绕过审计。
- 权限变化后继续使用无限期缓存。
- 只测试允许场景，不测试默认拒绝。

# 学习检查问题

1. 角色和权限为什么不能完全等同？
2. 项目角色为什么应放在成员关系而非 User？
3. 默认拒绝如何体现在设计与测试中？
4. 权限缓存键为什么需要作用域？
5. 什么时候角色数量增长说明模型设计有问题？

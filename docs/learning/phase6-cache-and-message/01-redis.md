> 主题：Redis 数据模型、过期、持久化与使用边界。

# 背景

数据库承担可靠持久化，但高频读取、短期状态和消息协调若全部访问 MySQL，会增加延迟与负载。Redis 用内存数据结构提供低延迟操作，适合缓存、计数、会话控制和消息 Broker。

它不是“更快的 MySQL”。内存成本、过期、淘汰和故障语义与关系数据库不同，误用会把性能优化变成正确性风险。

# 核心思想

Redis 以 Key 映射不同数据结构，单条命令通常原子执行。正确设计从访问模式、生命周期和一致性要求出发，而不是把任意对象 JSON 化后永久保存。

核心业务事实仍以 MySQL 为权威；Redis 中的数据要么可重建，要么有明确持久化与丢失容忍策略。

# 工作原理

Redis 主要在事件循环中处理命令，避免复杂锁竞争；网络、持久化和后台任务仍会影响延迟。String、Hash、Set、Sorted Set、Stream 等结构对应不同操作语义。

TTL 使 Key 到期，内存压力下淘汰策略可能提前删除数据。RDB 与 AOF 提供不同持久化取舍，但都不能自动赋予缓存数据库级业务一致性。

Lua 或事务命令可把一组 Redis 操作原子化，但无法与 MySQL 形成天然单事务。

# 企业实践

- Key 包含应用、环境、业务域和版本，设置合理 TTL。
- 连接池、超时和最大重试有明确配置。
- 大 Key、热 Key 和无界集合需要监控与治理。
- 缓存故障时核心业务优先回源或受控失败，不能默认授权。
- 分布式锁只用于明确互斥问题，并处理租约、超时和所有权。
- 不把 Redis `KEYS` 等阻塞性全量操作用于生产请求链路。

# TaskFlow Pro应用

Redis 可缓存用户权限、热点项目摘要和统计；保存 Token 撤销或会话版本；作为 Celery Broker。每类用途使用独立前缀和可观测指标。

权限缓存丢失时回源 MySQL，绝不默认允许。通知未读计数可缓存，但权威通知记录仍在数据库，缓存可重新计算。

# 常见误区

- 所有查询都缓存，忽略命中率和一致性成本。
- Key 永不过期。
- Redis 不可用时跳过权限检查。
- 用分布式锁替代数据库唯一约束。
- 把大列表作为一个 JSON String 反复整体读写。

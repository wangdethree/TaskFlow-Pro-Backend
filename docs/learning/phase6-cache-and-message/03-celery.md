> 主题：Celery Broker、Worker、任务重试与幂等。

# 背景

邮件、统计、文件处理和 LLM 调用耗时且易受外部服务影响。放在 HTTP 请求中会增加延迟、占用连接，并使临时故障直接表现为用户请求失败。

Celery 将工作放入队列，由独立 Worker 异步执行，实现请求与耗时任务解耦。

# 核心思想

生产者发布任务消息，Broker 保存和投递，Worker 获取并执行，结果后端可选。队列提供时间和资源上的解耦，不提供业务正确性。

消息常见语义是至少一次：任务可能因确认丢失或 Worker 崩溃重复执行。因此任务必须幂等，重试必须区分临时与永久错误。

# 工作原理

调用任务时，Celery 序列化任务名和参数发送到 Broker。Worker 预取消息并执行注册任务，根据确认策略在执行前后 ACK。失败可按退避和次数重试。

Worker 并发模型影响资源隔离；长任务会占用槽位，预取过多会造成不公平。任务参数应传稳定 ID，不传 ORM 对象或大文件内容，Worker 用自己的 Session 重新加载状态。

# 企业实践

- 任务小而有边界，设置超时、重试上限和退避。
- 使用任务/业务唯一 ID 实现幂等。
- 区分队列，将长 AI 任务与短通知任务隔离。
- 监控排队时间、执行时间、失败率和重试数。
- 失败任务进入可观察处理流程，不无限循环。
- 部署时确保任务代码版本与消息兼容。

# TaskFlow Pro应用

通知邮件、项目统计、孤儿文件清理和 AI 报告由 Celery 执行。HTTP 接口只返回已接受的任务标识或核心业务结果。

Worker 接收 task/project ID，重新加载数据并检查当前状态。通知用事件唯一键防重复，AI 队列独立设置更长超时和并发限制。

# 常见误区

- 把调用 `.delay()` 当作任务一定成功。
- 任务参数传递 AsyncSession 或 ORM 对象。
- 所有异常都无限重试。
- 没有幂等保护却开启自动重试。
- Web 与 Worker 版本不兼容仍消费旧消息。

# 学习检查问题

1. Producer、Broker、Worker 和结果后端分别负责什么？
2. 为什么任务可能重复执行？
3. 什么错误适合重试，什么错误应立即失败？
4. 为什么任务参数应传 ID 而非 ORM 对象？
5. AI 与通知任务为何可能需要不同队列？

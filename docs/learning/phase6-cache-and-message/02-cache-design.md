> 主题：Cache Aside、一致性、穿透、击穿与雪崩。

# 背景

缓存能降低读取延迟，却引入第二份数据。更新数据库后缓存仍是旧值、缓存失效时请求同时回源、恶意查询不存在数据，都会造成错误或数据库压力。

缓存设计首先是数据一致性和容量问题，其次才是 Redis API 使用问题。

# 核心思想

Cache Aside 中应用先读缓存，未命中则读数据库并回填；写入通常先更新数据库，再删除缓存。缓存是可丢弃派生数据，数据库是权威来源。

是否缓存取决于读取频率、计算成本、变化频率、容忍陈旧程度和命中率。不能接受短暂陈旧的授权撤销，需要更强失效或直接查权威数据。

# 工作原理

缓存未命中会回源。大量不存在 Key 形成穿透；热点 Key 到期瞬间的并发回源形成击穿；大量 Key 同时失效形成雪崩。

常见手段包括负缓存、布隆过滤、互斥重建、逻辑过期、随机化 TTL 和限流。每种手段都增加复杂度，要基于真实流量选择。

更新数据库后删除缓存仍有竞态窗口。延迟双删、订阅变更或版本化 Key 能降低风险，但没有通用零成本强一致方案。

# 企业实践

- 缓存前定义 Key、Value、TTL、失效事件和降级行为。
- 指标至少包括命中率、延迟、回源量、错误和内存。
- TTL 加抖动，避免批量同刻过期。
- 防止缓存空对象永远遮蔽新数据。
- 权限与会话缓存使用短 TTL、版本号或主动失效。
- 先通过性能数据证明需要缓存，不把缓存当默认层。

# TaskFlow Pro应用

项目详情可采用 Cache Aside，项目或成员变化后删除对应 Key。权限 Key 可包含用户、项目和权限版本，角色更新后提升版本或主动失效。

任务列表过滤组合多、变化频繁，早期不一定适合逐组合缓存；先优化索引和分页，再根据热点统计决定。

# 常见误区

- 数据库写成功后只更新缓存，不更新权威模型。
- 所有 Key 使用相同固定 TTL。
- 缓存空值却不设短过期。
- 用缓存掩盖缺失索引和 N+1。
- 没有命中率指标就声称性能提升。

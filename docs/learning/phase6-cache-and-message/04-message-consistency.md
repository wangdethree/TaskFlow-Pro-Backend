> 主题：数据库与消息的一致性、Outbox、幂等消费和补偿。

# 背景

业务事务与消息 Broker 属于不同系统。先提交数据库再发消息，进程可能在两步之间崩溃；先发消息再提交，消费者可能读不到数据或业务最终回滚。

消息一致性要处理的不是“调用顺序写对”，而是不可避免的部分失败。

# 核心思想

核心事实与待发布事件应尽可能在同一本地事务中记录。Outbox Pattern 把事件写入数据库表，独立发布器可靠投递，缩小数据库与 Broker 之间的不一致窗口。

投递仍可能重复，因此生产和消费两端都需要唯一标识、状态记录和幂等。最终一致性意味着短时间不一致可接受，但必须可检测、可重试和可补偿。

# 工作原理

业务事务写领域数据与 Outbox 行后提交。发布器扫描未发布事件，发送到 Broker，并更新发布状态。若发送成功但状态更新失败，会再次发送；消费者依靠 event ID 或业务唯一键去重。

消费者执行本地事务后确认消息。若在提交后、ACK 前崩溃，Broker 会重投，幂等逻辑保证结果不重复。死信或失败表承接长期失败供人工处理。

# 企业实践

- 事件包含唯一 ID、类型、聚合 ID、版本和发生时间。
- Payload 只放消费者需要的稳定数据，敏感信息最小化。
- 发布器支持并发领取、重试、告警和积压监控。
- 消费者先检查幂等记录，再与业务写入同事务完成。
- 定义事件 Schema 兼容策略，避免发布者升级破坏旧 Worker。
- 定期清理已发布记录但保留必要审计窗口。

# TaskFlow Pro应用

任务完成事务同时写 `TaskCompleted` Outbox 事件。Celery 发布通知和统计更新；重复消息由事件 ID 防止生成多条相同通知。

AI 报告任务可记录请求状态与幂等键，Worker 重试时更新同一报告而不是创建多个结果。长时间失败进入可查询失败状态。

# 常见误区

- 认为数据库提交后立即 `.delay()` 就绝不丢消息。
- 追求“恰好一次”口号却没有业务幂等。
- 消费成功后先 ACK，再提交数据库。
- 事件携带整个数据库对象快照。
- 没有积压和死信监控。
